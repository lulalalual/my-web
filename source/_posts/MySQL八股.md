MySQL 面试题看起来很多，底层其实围绕几条主线展开：数据怎么存，索引怎么查，事务怎么保证一致，锁怎么处理并发，日志怎么保证崩溃恢复。把这几条线串起来，比单独背答案更稳。

## InnoDB 为什么常用

InnoDB 是 MySQL 最常用的存储引擎。它支持事务、行级锁、外键和崩溃恢复。相比早期 MyISAM，InnoDB 更适合需要可靠性和并发写入的业务系统。

可以把存储引擎理解成数据库真正干活的底层模块。SQL 层负责解析语句和优化执行计划，存储引擎负责把数据读出来、写进去、加锁和维护索引。

## B+ 树为什么适合做索引

数据库索引常用 B+ 树，是因为它非常适合磁盘场景。普通二叉树层数可能很深，磁盘读取次数多；B+ 树每个节点能放很多 key，树高低，查一次数据需要的磁盘 IO 更少。

B+ 树的叶子节点按顺序连接，所以范围查询也方便。比如查 <code>id between 100 and 200</code>，找到起点后沿着叶子节点往后走就行。

聚簇索引里，叶子节点保存整行数据；普通二级索引里，叶子节点保存主键值，再通过主键回表查整行。这就是为什么覆盖索引能减少回表。

## 为什么主键建议自增

自增主键能让新数据大多插入到 B+ 树末尾，减少页分裂和数据搬移。如果使用随机 UUID 做主键，新数据可能插入到树的任意位置，更容易导致页分裂，写入成本更高。

当然，分布式系统可能需要全局唯一 ID，这时可以用雪花算法、号段模式等方案。工程里不是永远只能自增，而是要理解不同 ID 方案对索引写入的影响。

## 事务的 ACID 怎么理解

原子性表示一组操作要么都成功，要么都失败。转账时扣钱和加钱必须一起完成。

一致性表示事务前后数据规则不能被破坏。比如余额不能无缘无故变负，外键关系不能乱。

隔离性表示多个事务并发执行时，彼此影响要被控制。隔离级别越高，看到的数据越稳定，但并发成本也可能越高。

持久性表示事务提交后，即使数据库崩溃，数据也应该能恢复。这主要依赖 redo log。

## MVCC 解决了什么

MVCC 让读操作可以看到一个稳定的快照，而不是和写操作互相堵住。InnoDB 会通过隐藏字段、undo log 和 Read View 判断某个版本对当前事务是否可见。

你可以把它想成每行数据背后有一条版本历史。事务读取时，不是简单拿最新值，而是拿对自己可见的那个版本。这样就能在高并发下兼顾一致性和性能。

## undo log、redo log、binlog 分别做什么

undo log 用来回滚和提供历史版本。事务失败时，可以根据 undo log 把数据恢复到修改前。

redo log 用来保证崩溃恢复。事务提交后，即使数据页还没真正刷到磁盘，只要 redo log 落盘，数据库重启后就能重放恢复。

binlog 是 MySQL Server 层的日志，主要用于主从复制和数据恢复。它记录的是逻辑操作，比如某条 SQL 或某行数据变化。

面试时不要把三种日志混在一起。undo 关注撤销和版本，redo 关注崩溃恢复，binlog 关注复制和归档。

## 锁是为了保护并发修改

共享锁允许多个事务一起读，排他锁用于写。InnoDB 常见的是行锁，但行锁依赖索引。如果更新条件没有走索引，可能会扫描更多数据，锁范围也会变大。

间隙锁用于锁住一个范围之间的空隙，防止其他事务插入新记录。它主要和可重复读下的幻读控制有关。

锁问题排查时，先看事务是否太长，再看 SQL 是否走索引，最后看多个事务加锁顺序是否一致。

## 一条查询大概怎么执行

客户端发送 SQL 后，MySQL 会先解析语句，再由优化器选择执行计划，然后调用存储引擎读取数据。存储引擎可能走索引，也可能全表扫描。数据返回后，Server 层再做过滤、排序、聚合等操作。

理解这条路径后，优化方向也很清楚：让优化器能选到好索引，让存储引擎少读数据，让 Server 层少排序和少构造临时表。

## 面试复习方法

不要按题库机械背。可以用五个问题串起来：这张表的数据怎么存？查询为什么能快？多个事务一起改会怎样？崩溃后怎么恢复？主从怎么保持一致？

能围绕这五个问题解释清楚，MySQL 八股就会从零散答案变成一套体系。
