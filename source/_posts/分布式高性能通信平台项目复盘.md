分布式高性能通信平台可以理解成“RPC 框架 + IM 业务”的组合。它一边展示底层网络通信能力，一边用聊天业务验证这套通信能力真的能跑在业务场景里。

## 项目先说人话

单机程序里，函数调用很简单：A 调 B，结果马上返回。分布式系统里，A 和 B 可能在两台机器上，中间要经过网络、序列化、服务发现、超时和错误处理。

这个项目做的就是把这些细节包起来，让远程调用尽量像本地调用一样使用，同时再用 IM 聊天系统验证它。

~~~text
客户端请求 -> Muduo 收连接 -> 协议解析 -> RPC/IM 业务处理 -> Redis/MySQL/ZooKeeper 协作 -> 返回结果
~~~

## 技术栈

项目主要使用这些技术：

- C++17：使用现代 C++ 写网络和业务模块。
- Muduo：基于 Reactor 模型处理 TCP 连接和事件循环。
- Protobuf：负责 RPC 参数和结果的序列化。
- ZooKeeper：做服务注册与发现，用临时节点表达服务存活。
- Redis Pub/Sub：解决多台聊天服务器之间的消息推送。
- MySQL：保存用户、好友、群组和离线消息。
- nlohmann/json：处理 IM 消息里的 JSON 数据。
- CMake：管理多模块构建。

这些组件分工明确：Muduo 管连接，Protobuf 管数据格式，ZooKeeper 管服务在哪，Redis 管跨节点通知，MySQL 管长期数据。

## 整体架构

项目可以分成三层：公共基础层、RPC 框架层、IM 业务层。

公共基础层包括配置和日志。配置负责读取服务 IP、端口、ZooKeeper 地址等参数；日志负责把运行过程写下来，方便排查问题。

RPC 框架层包括 <code>RpcProvider</code>、<code>MprpcChannel</code> 和 <code>ZkClient</code>。服务端发布服务，客户端发现服务并发起调用，ZooKeeper 负责中间的地址查找。

IM 业务层包括 <code>ChatService</code>、用户模型、好友模型、群组模型、离线消息模型和 Redis 发布订阅。

~~~text
common/config/logger
        ↓
rpc/provider/channel/zkclient
        ↓
im/chatservice/model/redis
~~~

## RPC 协议怎么设计

RPC 最核心的问题是：网络上收到一串字节，怎么知道它属于哪个服务、哪个方法、参数有多长。

项目使用了一个自定义协议：

~~~text
4字节 header_size + protobuf header + protobuf args
~~~

header 里保存 service_name、method_name 和 args_size。接收端先读 4 字节知道 header 长度，再读 header，最后读参数。这样能解决 TCP 粘包和半包问题。

这也是 RPC 项目里最值得讲的点之一：TCP 只负责字节流，不负责消息边界；应用层必须自己定义边界。

## 服务发布和服务发现

服务端通过 <code>RpcProvider</code> 发布服务。业务类继承 Protobuf 生成的 service 基类，实现具体方法，然后调用 NotifyService 注册到框架。

服务启动后，会把服务名和方法名注册到 ZooKeeper，例如：

~~~text
/dhc_comm_service/UserService/Login -> 192.168.1.100:9000
~~~

客户端调用时先问 ZooKeeper：我要调用 UserService.Login，地址在哪？拿到地址后再直连目标服务。

ZooKeeper 使用临时节点的好处是：服务挂了，连接断开，节点会自动消失，客户端就不会一直拿到一个坏地址。

## IM 业务怎么跑

IM 业务处理的是登录、注册、单聊、加好友、建群、群聊、注销这些消息。

可以把单聊流程理解成：

~~~text
用户 A 发消息 -> ChatService 查用户 B 是否在线 -> 在线就转发 -> 不在线就存离线消息
~~~

如果用户 B 连接在另一台服务器上，就不能只查本机连接表。这时 Redis Pub/Sub 派上用场：A 所在服务器把消息发布到 B 的频道，B 所在服务器订阅频道并把消息推给用户。

~~~text
Server A Publish(user_456) -> Redis -> Server B Subscribe(user_456) -> 推给用户 456
~~~

这解决了多服务器扩展时的一个关键问题：用户不一定在同一台机器上。

## 数据怎么落库

MySQL 负责保存用户、好友、群组、群成员和离线消息。业务层不直接拼一堆 SQL，而是通过 UserModel、FriendModel、GroupModel、OfflineMsgModel 这类模型封装数据库操作。

这种拆法的好处是 ChatService 只关心业务流程，不需要关心每张表怎么写 SQL。

## 日志和配置

分布式项目出问题时，日志非常重要。比如服务有没有注册成功，客户端查到的地址是什么，RPC 请求有没有反序列化失败，Redis 订阅线程有没有断开。

配置同样重要。IP、端口、ZooKeeper 地址、数据库连接不应该写死在代码里，否则换环境就要改源码。

## 难点和踩坑

第一个难点是协议边界。很多新手写网络程序时会以为一次 send 对应一次 recv，但 TCP 不是这样。项目通过 header_size 明确消息长度，这是解决粘包问题的关键。

第二个难点是服务发现。服务不是永远在线的，客户端不能写死地址。ZooKeeper 临时节点能表达服务健康，但客户端也要考虑节点变更、连接失败和重试。

第三个难点是跨服务器消息。单机聊天只要查本机连接表，多机聊天必须解决“目标用户在哪台服务器”。Redis Pub/Sub 是一个简单有效的方案。

第四个难点是模块边界。RPC 框架和 IM 业务结合时，不能互相污染太多。框架应该可复用，业务应该可替换。

## 如果继续优化

后续可以继续补这些方向：

- 给 RPC 增加超时、重试和熔断。
- 增加负载均衡，不只拿一个服务地址。
- Redis Pub/Sub 增加断线重连和消息可靠性补偿。
- 给 IM 增加消息 ACK，减少丢消息风险。
- 加压测，分别测连接数、QPS、延迟和数据库瓶颈。
- 增加更清晰的监控指标，比如在线人数、消息队列长度、RPC 错误率。

## 项目总结

这个项目最有价值的地方是把底层框架和真实业务放在一起。RPC 展示网络、协议、序列化和服务发现能力；IM 展示用户、好友、群组、离线消息和跨服务器推送。两者结合起来，比单独写一个 Demo 更能说明工程理解。
