塔防双人是一个用 C++ 和 SDL2 写的桌面塔防游戏。它的价值不只是能运行，而是把一个游戏拆成了多个清楚的工程模块。

## 项目先说人话

塔防游戏本质上是一个持续运行的模拟系统。每一帧都要读玩家输入、更新所有对象状态、再把结果画出来。

~~~text
处理输入 -> 更新玩家/敌人/炮塔/子弹 -> 碰撞和伤害结算 -> 绘制画面
~~~

主循环就是整个游戏的心跳。只要主循环稳定，地图、怪物、炮塔、子弹这些模块就能按节奏协作。

## 技术栈

项目使用 C++ 和 SDL2 系列库实现，主要包括：

- SDL2：窗口、事件和基础渲染。
- SDL_image：加载图片资源。
- SDL_mixer：播放音效和音乐。
- SDL_ttf：渲染文字。
- SDL2_gfx：补充图形效果。
- cJSON：读取配置文件。
- Visual Studio 工程：用 sln/vcxproj 管理构建。

入口很简单，<code>main.cpp</code> 最后调用 <code>GameManager::instance()->run()</code>。这说明项目把启动、循环和退出逻辑集中到了 GameManager 里，而不是散在 main 函数中。

## 工程模块怎么拆

项目目录里能看到很多以 manager 结尾的文件，这是一种很直观的拆法：每个管理器负责一类对象。

- <code>game_manager.h</code>：控制游戏整体生命周期和主循环。
- <code>resources_manager.h</code>：统一加载图片、字体、音效等资源。
- <code>config_manager.h</code>：读取窗口、玩家、炮塔、敌人配置。
- <code>player_manager.h</code>：处理玩家移动、攻击和技能。
- <code>enemy_manager.h</code>：生成和更新敌人。
- <code>tower_manager.h</code>：管理炮塔建造、升级和攻击。
- <code>bullet_manager.h</code>：管理子弹生成、移动和命中。
- <code>coin_manager.h</code>：管理金币掉落和拾取。
- <code>wave_manager.h</code>：控制波次节奏。

这种拆分的好处是后面改某个功能时，不需要在一个巨大文件里到处找。

## 配置驱动

游戏数值放在 <code>config.json</code> 里，比如玩家速度、普通攻击间隔、技能伤害、炮塔攻击间隔、炮塔升级成本、怪物血量和速度。

这比把数值写死在 C++ 代码里更好。因为调平衡时可以直接改配置，不用重新理解业务代码。

~~~json
{
  "player": {
    "speed": 5,
    "normal_attack_damage": 10,
    "skill_damage": 20
  }
}
~~~

塔防游戏非常依赖数值平衡。怪物太肉会劝退，炮塔太强会没挑战，所以配置化是很实用的工程设计。

## 地图和路线

塔防的核心是路径。怪物不是乱走，而是沿着路线前进。项目里有 <code>map.csv</code>、<code>map.h</code>、<code>route.h</code> 和 <code>tile.h</code>，说明地图数据、路线和格子被拆开管理。

可以这样理解：

~~~text
map.csv 定义格子 -> Map 读取地图 -> Route 提供路径点 -> Enemy 按路径移动
~~~

这种设计让地图和怪物解耦。怪物只需要知道下一段路在哪里，不需要关心整张地图怎么存。

## 敌人系统

项目里有多种敌人，比如 slim、king_slim、skeleton、goblin、goblin_priest。不同敌人有不同血量、速度、伤害、奖励和恢复能力。

这体现了继承或多态的使用场景：基础 Enemy 负责通用行为，具体敌人负责差异能力。

比如牧师类怪物可以有恢复范围和恢复强度，普通怪物则只需要移动和扣血。这样设计后，新增敌人时只需要补具体类，不用重写整个敌人系统。

## 炮塔和子弹

炮塔也分类型，比如弓箭塔、斧头塔、枪塔。每种塔有攻击间隔、伤害、范围、建造成本和升级成本。

炮塔本身不一定直接处理所有命中逻辑，更合理的链路是：

~~~text
Tower 发现目标 -> BulletManager 生成子弹 -> Bullet 飞行 -> 命中 Enemy -> 结算伤害
~~~

这样炮塔负责“什么时候打、打谁”，子弹负责“怎么飞、什么时候命中”，敌人负责“扣血和死亡”。职责分开后，代码更容易扩展。

## 玩家系统

这个项目不是纯自动塔防，还加入了玩家。玩家有移动速度、普通攻击、技能间隔和技能伤害。这会让游戏更像“塔防 + 角色协作”。

玩家系统的难点是输入和状态同步。比如按键移动、攻击冷却、技能冷却、动画方向、和怪物距离判断，都要在每一帧更新。

## 资源管理

游戏资源包括图片、音效、字体和动画。如果每个对象自己加载资源，会造成重复加载和释放混乱。统一资源管理器能避免这个问题。

资源管理器的职责是：加载一次，多处使用，退出时统一释放。这其实就是游戏项目里的 RAII 思想：资源生命周期必须有人管。

## 难点和踩坑

第一个难点是主循环节奏。如果 update 和 render 混在一起，帧率不稳定时游戏速度会忽快忽慢。所以需要用 deltaTime 或定时器控制更新。

第二个难点是对象生命周期。子弹命中后要删除，怪物死亡后要发奖励，金币拾取后要消失。如果删除时机不对，容易出现野指针或遍历容器时失效。

第三个难点是数值平衡。塔防游戏不是把功能做出来就结束，还要反复调怪物血量、炮塔伤害、波次节奏和金币收益。

## 如果继续优化

后续可以继续补这些能力：

- 加关卡编辑器，让地图不再手写 CSV。
- 增加寻路算法，支持更复杂路径。
- 做对象池，减少子弹和特效频繁创建销毁。
- 增加双人协作逻辑，比如分工、共享资源和技能配合。
- 把 UI 面板和游戏逻辑进一步解耦。

## 项目总结

塔防双人最能体现的是 C++ 游戏工程拆分能力。它把主循环、地图、路线、敌人、炮塔、子弹、玩家、资源和配置拆成独立模块，既能练面向对象，也能练资源管理和实时系统思维。
