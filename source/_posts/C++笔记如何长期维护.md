C++ 笔记最容易写成两种样子：一种是复制定义，过几天自己也看不懂；另一种是只记结论，遇到代码还是不会用。真正有用的笔记，应该能帮你重新理解问题，并且能直接回到代码里。

## 先说人话

笔记不是收藏夹，也不是百科。它应该回答三个问题：这个知识点解决什么问题，代码里怎么用，容易在哪里出错。

~~~text
问题 -> 解释 -> 例子 -> 边界 -> 复盘
~~~

如果一篇笔记只有概念，没有例子；或者只有代码，没有解释，复习时都会很吃力。

## 标题要短，问题要明确

标题不要写得太大。比如“C++ 高级知识大全”很难维护，因为它什么都想装。更好的标题是“智能指针”“移动语义”“线程同步”。

标题短，不代表内容浅，而是让文章边界清楚。

一个好标题下面应该能接一句问题：

~~~text
智能指针：谁负责释放资源？
移动语义：怎么少做一次昂贵拷贝？
线程同步：多个人改同一份数据怎么不乱？
~~~

标题越具体，写作越不容易跑偏。

## 每篇都要先写一句人话

很多概念一上来就背定义，会把自己绕住。比如 RAII，可以先写成：资源跟着对象走，对象活着资源就在，对象销毁资源就释放。

然后再补专业解释：RAII 是 Resource Acquisition Is Initialization，通过构造函数获取资源，通过析构函数释放资源，利用对象生命周期保证异常安全。

先人话，后术语，这样复习时不会只有空壳。

## 代码例子要小

笔记里的例子不要一上来写完整项目。最小例子更适合复习，因为它只突出一个问题。

比如解释引用：

~~~cpp
void addOne(int& x) {
    ++x;
}

int value = 1;
addOne(value);
~~~

这个例子很小，但它能说明引用会直接影响原对象。后面再扩展到函数参数、返回值和生命周期问题。

## 必须写常见坑

C++ 的难点不是语法本身，而是边界。每篇笔记都应该有“常见坑”。

比如智能指针：

- <code>shared_ptr</code> 循环引用会导致对象无法释放。
- 不要用同一个裸指针构造多个 <code>shared_ptr</code>。
- <code>unique_ptr</code> 不能拷贝，只能移动。
- 智能指针不是所有权问题的万能答案。

这些坑比定义更能帮你写对代码。

## 要把知识点放进场景

如果只写“mutex 用来加锁”，还是太抽象。应该继续写：当多个线程同时修改同一个变量时，会出现数据竞争，所以要用 mutex 把临界区保护起来。

~~~cpp
std::mutex m;
int count = 0;

void add() {
    std::lock_guard<std::mutex> lock(m);
    ++count;
}
~~~

场景越具体，知识点越容易记住。

## 重复内容要合并

同一个知识点散在五篇文章里，后面一定会乱。比如指针、引用、const、生命周期这些内容，本质都属于 C++ 基础，应该合并到一篇基础长文里。

重复内容可以这样处理：

- 基础概念放到主文章。
- 项目里遇到的坑放到项目复盘。
- 面试问法放到八股总结。
- 只保留一个权威入口，其他地方链接过去。

这样维护成本会低很多。

## 不要只写“正确答案”

笔记里可以保留自己踩坑的过程。比如“我一开始以为 vector erase 后迭代器还能继续用，后来发现会失效”。这种记录很有价值，因为它说明这个坑真的影响过代码。

复盘时可以按这个格式写：

~~~text
我以为：erase 后继续用旧迭代器没事
实际：迭代器可能失效
正确做法：使用 erase 返回的新迭代器继续遍历
~~~

这种写法比单纯背规则更容易记住。

## 面试答案要能收放

同一个知识点，面试时不能一口气背三分钟。笔记里可以准备三层答案：一句话版、展开版、追问版。

比如 “vector 和 list 区别”：

一句话版：vector 连续内存，随机访问快；list 链式存储，已知位置插删方便。

展开版：vector 中间插删要移动元素，但缓存友好；list 查找慢，节点额外存指针，缓存局部性差。

追问版：实际工程里多数情况优先 vector，因为 CPU 缓存收益很大，list 不是插删多就一定更好。

## 定期清理

笔记不是写完就结束。隔一段时间要回来看：有没有重复，有没有过时，有没有例子不清楚，有没有和项目经验脱节。

建议维护节奏：

- 每周修一篇表达不清楚的文章。
- 每次项目踩坑后补一段复盘。
- 每次面试或刷题后补“追问点”。
- 每月合并重复短文。

## 总结

C++ 笔记的目标不是显得专业，而是让自己下次能更快讲清楚、更快写对代码。好笔记应该有问题、有例子、有边界、有坑、有复盘。只要按这个结构长期维护，笔记会越来越像自己的技术地图。
