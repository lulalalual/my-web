SQL 进阶不是写更复杂的语句，而是理解数据库为什么这样执行。基础阶段关心“能不能查出来”，进阶阶段要关心“查得快不快、锁会不会冲突、数据会不会一致”。

## 执行计划告诉你数据库怎么干活

同一条 SQL，数据库可能有很多执行方式。它可以全表扫描，也可以走索引；可以先过滤再排序，也可以先关联再过滤。执行计划就是数据库给出的“工作路线”。

~~~sql
EXPLAIN SELECT *
FROM users
WHERE email = 'a@example.com';
~~~

看执行计划时，先关注三件事：有没有用到预期索引，扫描行数大不大，是否出现额外排序或临时表。不要只看 SQL 短不短，短 SQL 也可能很慢。

## 联合索引要按最左前缀理解

联合索引像一本按多列排序的通讯录。比如索引是 <code>(city, age, name)</code>，它会先按城市排，再按年龄排，最后按名字排。

~~~sql
CREATE INDEX idx_city_age_name ON users(city, age, name);
~~~

如果查询条件里有 <code>city</code>，就能很好利用索引；如果跳过 <code>city</code> 只查 <code>age</code>，索引效果就会差很多。这就是最左前缀原则。

设计联合索引时，应该把区分度高、使用频繁、常用于等值查询的字段放在前面。不是字段越多越好，索引越宽，维护成本越高。

## 覆盖索引可以减少回表

如果查询需要的列都在索引里，数据库就不必再回到原表拿数据，这叫覆盖索引。

~~~sql
CREATE INDEX idx_user_status_time ON orders(user_id, status, created_at);

SELECT user_id, status, created_at
FROM orders
WHERE user_id = 10 AND status = 'paid';
~~~

这条查询只需要索引里的字段，就可能直接从索引返回结果。覆盖索引常用于高频列表页、统计页和接口热点查询。

## 事务隔离级别是在平衡一致性和并发

多个事务同时执行时，会出现脏读、不可重复读、幻读等问题。隔离级别越高，数据越安全，但并发能力可能越低。

读未提交几乎不适合普通业务，因为它可能读到别人还没提交的数据。读已提交能避免脏读。可重复读能让同一个事务里多次读取结果保持一致。串行化最严格，但并发最差。

MySQL InnoDB 默认使用可重复读，并通过 MVCC 和锁机制处理一致性问题。面试里不要只背四个隔离级别，最好能解释为什么业务需要在一致性和性能之间取舍。

## MVCC 让读写尽量互不阻塞

MVCC 可以理解成给数据保存多个版本。读事务看到的是某个时间点的快照，写事务可以继续修改新版本。这样读和写不一定互相堵住。

这就像你在图书馆看一本书的复印件，管理员同时可以更新原书。你读到的是开始阅读那一刻的版本，不会被中途修改打断。

MVCC 不等于没有锁。更新同一行时仍然需要锁，范围查询也可能涉及间隙锁。它解决的是一致性读的并发问题，不是所有并发问题。

## 慢查询优化要先定位瓶颈

优化 SQL 时不要上来就加索引。先用慢查询日志或执行计划定位问题。常见原因包括：没有索引、索引失效、扫描行数太多、返回列太多、JOIN 顺序不合理、深分页、函数包住索引列。

~~~sql
SELECT *
FROM users
WHERE DATE(created_at) = '2026-04-30';
~~~

这类写法可能让索引失效，因为数据库需要先对每一行做函数计算。更好的写法是改成范围查询。

~~~sql
SELECT *
FROM users
WHERE created_at >= '2026-04-30 00:00:00'
  AND created_at <  '2026-05-01 00:00:00';
~~~

优化的核心是让数据库少扫数据、少排序、少回表、少传输。

## 分库分表不是第一选择

当单表数据越来越大时，很多人第一反应是分库分表。但分库分表会带来跨库查询、事务、分页、扩容和运维复杂度。它不是优化的起点，而是单机和索引优化都接近瓶颈后的工程方案。

在考虑分库分表之前，先确认表结构是否合理，索引是否有效，冷热数据是否能归档，读写是否能分离，缓存是否能承接热点。

## 进阶 SQL 的学习标准

如果你能看到一条慢 SQL，就判断它为什么慢；看到一个索引设计，就说出它服务哪些查询；看到一个事务问题，就能解释隔离级别、MVCC 和锁的关系，那么 SQL 就不再只是语法，而是工程能力。
