网络编程的本质，是让两台机器上的程序通过约定好的协议交换数据。不要一上来就背 socket API，先把通信过程想成打电话：一方监听电话，一方拨号，接通后双方按约定格式说话，最后挂断。

## TCP 和 UDP 先按场景区分

TCP 像打电话，连接建立后再交流。它保证数据有序、可靠，适合文件传输、登录、聊天、数据库连接等场景。UDP 像发明信片，不先建立连接，也不保证一定送到，但开销小，适合实时音视频、游戏状态同步、广播发现等场景。

选择协议时，不要只问哪个更快，而要问业务更怕什么。怕丢数据，就优先 TCP；怕延迟，就考虑 UDP 并在业务层补可靠性。

## 一个 TCP 服务端怎么工作

最简单的 TCP 服务端会经历几个步骤：创建 socket，绑定 IP 和端口，开始监听，接受连接，读写数据，关闭连接。

~~~cpp
int listenfd = socket(AF_INET, SOCK_STREAM, 0);
bind(listenfd, ...);
listen(listenfd, 128);
int connfd = accept(listenfd, ...);
read(connfd, buffer, sizeof(buffer));
write(connfd, response, len);
close(connfd);
~~~

这些函数名看起来多，但流程很自然。<code>socket</code> 是拿到一部电话，<code>bind</code> 是告诉别人拨哪个号码，<code>listen</code> 是开始等电话，<code>accept</code> 是接电话，<code>read/write</code> 是交流。

## 字节流没有天然消息边界

TCP 传的是字节流，不是“一条一条消息”。你发送两次，对方可能一次读到；你发送一次，对方也可能分多次读到。这就是常说的粘包和半包。

解决办法是在协议里自己定义边界。常见做法是在消息头里放长度，接收端先读固定长度的头，再按长度读正文。

~~~text
[4 字节长度][JSON 或二进制正文]
~~~

只要理解 TCP 是水管而不是快递盒，粘包问题就不神秘。水管只保证水按顺序流过去，不保证你倒进去的每一杯水在出口还保持杯子的形状。

## 阻塞、非阻塞和 IO 多路复用

阻塞 IO 很好理解：没有数据时，线程就停在那里等。连接少时这很简单，连接多时就会浪费大量线程。

非阻塞 IO 不会一直等，没有数据就返回。它需要配合 <code>select</code>、<code>poll</code>、<code>epoll</code> 这类 IO 多路复用机制，让一个线程同时管理很多连接。

<code>epoll</code> 可以理解成一个事件通知器。你把关心的 socket 注册进去，内核在它们可读或可写时通知你。这样线程不需要挨个问每个连接有没有数据。

## Reactor 是事件驱动模型

很多 C++ 网络库会使用 Reactor 模型。它的核心是：一个事件循环负责等事件，事件来了就分发给对应处理器。

~~~text
事件循环 -> 监听可读/可写 -> 分发回调 -> 处理业务 -> 继续等待
~~~

这种模型适合高并发连接，因为线程不需要为每个连接单独阻塞。Muduo 这类库就是围绕 Reactor 思路设计的。

## 协议设计比 API 更重要

能收发字节只是第一步。真正的工程问题是协议怎么设计：消息怎么分包，字段怎么兼容，错误码怎么表达，超时怎么处理，心跳怎么设计，旧客户端和新服务端能不能一起工作。

一个简单但清晰的协议通常比复杂但漂亮的协议更可靠。例如内部服务可以用 Protobuf 定义结构，再用长度头解决边界问题。外部接口可以用 HTTP 和 JSON，方便调试和扩展。

## 网络程序要考虑异常路径

网络不可靠是常态。客户端可能突然断开，半天不发数据，发来非法数据，网络抖动导致超时。服务端代码不能假设对方总是按规则来。

处理网络异常时，常见策略是：读到 0 表示对端关闭；读写错误要区分可重试和不可恢复；长时间无心跳就断开；单个连接的错误不能拖垮整个进程。

## 学习路线

先手写一个阻塞 TCP echo 服务，理解 socket 基本流程；再给消息加长度头，解决粘包半包；然后改成非阻塞和 epoll；最后学习 Reactor、线程池、连接管理和协议序列化。

网络编程不是 API 背诵题，而是对“连接、事件、协议、异常、并发”这五件事的系统理解。
