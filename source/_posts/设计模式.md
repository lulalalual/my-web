设计模式不是背模板，也不是把代码写得复杂。它们是前人在反复遇到类似问题后，总结出来的一些组织代码的方法。学设计模式时要先问：它解决什么变化？如果没有变化点，就不需要模式。

## 单例模式控制全局唯一对象

单例适合那些系统里确实只应该有一个实例的对象，比如配置中心、日志器。它的目标是限制创建数量，并提供统一访问点。

~~~cpp
class Logger {
public:
    static Logger& instance() {
        static Logger logger;
        return logger;
    }

    void info(const std::string& msg) {
        std::cout << msg << std::endl;
    }

private:
    Logger() = default;
};
~~~

现代 C++ 里，函数内静态变量初始化是线程安全的。单例不要滥用，因为全局状态会让测试和依赖关系变复杂。

## 工厂模式把创建过程收起来

如果创建对象的逻辑很复杂，或者调用方不应该关心具体类，就可以用工厂。调用方只说“我要一个对象”，工厂决定创建哪种。

~~~cpp
std::unique_ptr<Shape> createShape(const std::string& type) {
    if (type == "circle") return std::make_unique<Circle>();
    if (type == "rect") return std::make_unique<Rect>();
    return nullptr;
}
~~~

工厂模式的价值是隔离变化。以后新增类型时，调用方不必到处改 <code>new</code>。

## 策略模式把可变算法拆出去

当同一件事有多种做法时，可以用策略模式。比如排序规则、计费规则、推荐规则，都可能根据场景变化。

~~~cpp
class PriceStrategy {
public:
    virtual double calc(double price) const = 0;
    virtual ~PriceStrategy() = default;
};
~~~

调用方只依赖策略接口，具体使用满减、折扣还是会员价，可以运行时替换。策略模式比一长串 if-else 更容易扩展和测试。

## 观察者模式处理一对多通知

当一个对象状态变化后，需要通知多个对象，就可以考虑观察者模式。比如按钮点击后通知多个监听器，消息到达后通知多个订阅者。

它的好处是发布者不需要知道订阅者具体是谁，只要按接口通知即可。缺点是调用链可能变隐蔽，调试时要能追踪事件从哪里发出、被谁处理。

## 装饰器模式在不改原类的情况下加能力

装饰器像给对象套外壳。原对象负责核心功能，装饰器负责在前后增加额外行为，比如日志、缓存、权限校验。

这种模式适合功能可以层层叠加的场景。它比继承更灵活，因为你可以在运行时组合不同装饰器。

## 代理模式控制访问

代理对象站在真实对象前面，帮它做访问控制、延迟加载、远程调用或缓存。RPC 客户端里的 Stub 就很像代理：调用方以为在调用本地方法，实际上代理把请求发到了远程服务。

代理模式的关键是接口保持一致。调用方不需要知道背后是真实对象还是代理对象。

## 不要为了模式而模式

设计模式真正有价值的时候，是代码已经出现变化点：创建逻辑变多、算法经常换、通知对象变多、功能需要组合、访问需要控制。如果只是几行简单逻辑，强行套模式只会让代码更难读。

学习设计模式的正确方式，是先写直接代码，发现重复和变化后再抽象。模式是为代码服务的，不是代码必须服务模式。
