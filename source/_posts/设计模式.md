设计模式不是一套必须背下来的模板，也不是为了把简单代码写复杂。它更像是前人在反复写业务代码时总结出来的经验：当某种变化经常出现时，可以用一种更稳定的组织方式把变化关起来。

## 先理解模式解决什么变化

学习设计模式时，不要从名字开始背。你要先问：这里到底哪里会变？对象创建会变，就考虑工厂；算法规则会变，就考虑策略；一个对象变化后要通知很多人，就考虑观察者；想在不改原类的情况下加功能，就考虑装饰器。

如果代码没有变化点，强行套模式只会让代码更绕。模式不是高级写法的证明，而是管理变化的工具。

## 单例：控制全局唯一对象

单例模式适合系统里确实只应该有一个实例的对象，比如日志器、全局配置、资源管理器。它的重点不是“全局访问很方便”，而是“对象数量必须受控”。

~~~cpp
class Logger {
public:
    static Logger& instance() {
        static Logger logger;
        return logger;
    }

    void info(const std::string& msg) {
        std::cout << msg << std::endl;
    }

private:
    Logger() = default;
};
~~~

现代 C++ 中，函数内部的静态局部变量初始化是线程安全的，所以这种写法比手写双重检查锁更稳。需要注意的是，单例会带来全局状态，测试和依赖管理会变复杂，不要因为方便就到处使用。

## 工厂：把创建逻辑集中起来

当创建对象的逻辑开始变复杂，或者调用方不应该知道具体类名时，就可以使用工厂。调用方只告诉工厂“我要什么类型”，工厂负责创建具体对象。

~~~cpp
std::unique_ptr<Shape> createShape(const std::string& type) {
    if (type == "circle") return std::make_unique<Circle>();
    if (type == "rect") return std::make_unique<Rect>();
    return nullptr;
}
~~~

这段代码的好处是，调用方不再到处写 `new Circle`、`new Rect`。以后新增三角形，只需要扩展工厂或注册表，业务代码不用大面积修改。

## 策略：把可替换的算法拆出去

策略模式适合“同一件事有多种做法”的场景。比如价格计算可以有满减、折扣、会员价；排序可以按时间、热度、权重；游戏里敌人 AI 可以有巡逻、追击、逃跑。

~~~cpp
class PriceStrategy {
public:
    virtual double calc(double price) const = 0;
    virtual ~PriceStrategy() = default;
};

class DiscountStrategy : public PriceStrategy {
public:
    double calc(double price) const override {
        return price * 0.8;
    }
};
~~~

如果不用策略模式，代码很容易变成一长串 `if else`。策略模式把变化的规则拆到独立类里，调用方只依赖统一接口。以后换规则，不需要动主流程。

## 观察者：一个变化通知多个对象

观察者模式处理的是一对多通知。比如按钮被点击后，多个监听器都要响应；用户上线后，好友列表、消息系统和状态系统都要知道；游戏里怪物死亡后，金币系统、任务系统和音效系统都要收到消息。

观察者的优势是解耦。发布者只知道“我要发一个事件”，不需要知道谁会处理。订阅者只关心自己收到事件后做什么。缺点是调用链会变隐蔽，所以事件命名、日志和调试工具要做好。

## 装饰器：不改原类也能加能力

装饰器模式像给对象套一层外壳。原对象负责核心功能，装饰器在前后加日志、缓存、权限校验、重试等能力。

这种模式比继承灵活。继承会让类层次越来越深，而装饰器可以在运行时自由组合。例如一个网络请求可以先套日志，再套缓存，再套重试，每一层只做自己的事情。

## 代理：让访问先经过中间层

代理模式让调用方不直接接触真实对象，而是先经过代理。代理可以做权限控制、远程调用、延迟加载、缓存和监控。

RPC 客户端就是很典型的代理。调用方看起来像在调用本地函数，代理对象实际会把参数序列化，通过网络发给远程服务，再把结果反序列化回来。

## 模板方法：固定流程，开放步骤

模板方法适合流程稳定、某些步骤可变的场景。父类定义整体流程，子类只改其中几个步骤。比如导入文件的流程可以固定为读取、解析、校验、保存，但 CSV、JSON、Excel 的解析方式不同。

这种模式的价值是把主流程放在一个地方，避免每个子类都复制一遍流程。缺点是继承关系会变强，所以流程真的稳定时再用。

## 设计模式怎么落地

真正写项目时，不要一开始就套模式。先写直接代码，等变化点出现后再抽象。比如只有两个价格规则时，简单 `if` 可能够用；当规则变成十几个，还经常新增，策略模式就更合适。

判断一个模式该不该用，可以问三句话：它有没有隔离变化？有没有减少重复？有没有让调用方更清楚？如果只是让代码文件变多、概念变多，那就不要用。

设计模式学到最后，不是记住 23 个名字，而是形成一种习惯：看到变化，先把变化关到合适的位置；看到重复，先找出稳定流程；看到耦合，先设计更清楚的边界。
