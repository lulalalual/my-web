设计模式不是为了让代码看起来高级，而是为了解决反复出现的维护问题。学模式最重要的不是背名字，而是知道什么时候该用，什么时候不该用。

## 先说人话

模式大致分三类：

~~~text
创建型：对象怎么造
结构型：对象怎么组合
行为型：对象怎么协作
~~~

如果代码很简单，就不要硬套模式。模式的目标是让变化更容易处理，不是把简单代码写复杂。

## 为什么需要设计模式

写代码一开始通常很直接：需要什么就写什么。但项目变大后，会出现几类问题：

- 创建对象的逻辑到处散落。
- if-else 越来越多，新增功能容易改坏旧逻辑。
- 模块之间互相依赖，改一个地方牵动一大片。
- 事件通知混乱，一个状态变化要手动通知很多地方。

设计模式就是针对这些高频问题总结出来的经验。

## 单例：全局只需要一个

单例适合日志、配置、线程池这类全局唯一对象。

~~~cpp
class Logger {
public:
    static Logger& instance() {
        static Logger logger;
        return logger;
    }
    void log(const std::string& msg) {}
private:
    Logger() = default;
};
~~~

C++ 里函数局部静态变量的初始化是线程安全的，所以这种写法比较常见。

但单例也有代价。它容易变成全局变量换皮，让测试和依赖管理变困难。能通过参数传入的对象，不一定非要做成单例。

## 工厂：把创建细节收起来

当创建对象有很多分支时，工厂可以把 new 哪个类的逻辑统一起来。

~~~cpp
std::unique_ptr<Enemy> createEnemy(const std::string& type) {
    if (type == "fast") return std::make_unique<FastEnemy>();
    if (type == "tank") return std::make_unique<TankEnemy>();
    return std::make_unique<NormalEnemy>();
}
~~~

调用方只关心“我要什么类型”，不关心具体构造过程。

游戏项目里创建敌人、炮塔、子弹都适合用工厂。后面新增敌人类型时，修改集中在工厂里，而不是散落在多个地方。

## 策略：把可替换算法拆出去

策略模式适合“同一个动作有多种实现”的场景。比如支付方式、排序规则、攻击方式、评分规则。

~~~cpp
class AttackStrategy {
public:
    virtual int damage() const = 0;
    virtual ~AttackStrategy() = default;
};
~~~

如果不用策略，代码可能变成一长串 if-else：如果是弓箭塔这样打，如果是火炮塔那样打，如果是魔法塔再换一种。

策略模式把这些算法拆成独立类，让新增策略时尽量不改旧逻辑。

## 观察者：一处变化，多处响应

观察者适合事件通知。比如用户登录后要刷新 UI、写日志、同步状态；游戏里怪物死亡后要加金币、播放动画、更新任务。

发布者不需要知道谁在听，只负责发事件。订阅者自己决定怎么处理。

~~~cpp
class Observer {
public:
    virtual void onEvent(int value) = 0;
    virtual ~Observer() = default;
};
~~~

观察者的好处是解耦，但也容易让调用链变隐蔽。事件太多时，要有清楚的命名和日志，否则排查问题会麻烦。

## 适配器：接口对不上，包一层

适配器解决的是“旧接口和新系统不匹配”的问题。

~~~cpp
class PrinterAdapter {
public:
    explicit PrinterAdapter(OldPrinter& p) : printer(p) {}
    void print(const std::string& text) {
        printer.printOld(text);
    }
private:
    OldPrinter& printer;
};
~~~

适配器的核心是不改旧代码，也能让新接口使用它。

项目里接第三方库时经常会用到适配器。比如库的返回格式和自己系统不一致，就包一层转换。

## 装饰器：不改原类，加新能力

装饰器适合在原有功能外面加一层增强，比如日志、缓存、权限校验、性能统计。

人话理解：核心对象继续做核心事，增强功能放到外面包一层。

比如一个请求处理器本来只处理业务，外面可以包一层日志装饰器，记录请求耗时；再包一层权限装饰器，检查用户身份。

装饰器的价值是组合能力，但层数太多时也会增加阅读成本。

## 代理：访问前多一层控制

代理模式是在真实对象前面放一个代表。调用方以为自己在访问真实对象，其实先经过代理。

常见用途：

- 权限控制：没权限就不转发。
- 延迟加载：真正需要时再加载大对象。
- 远程调用：本地代理把请求发到远程服务。
- 缓存：先查缓存，没命中再访问真实对象。

RPC 里的 Stub 就有点像代理：你调用本地方法，底层其实发起了网络请求。

## 责任链：让请求一路传下去

责任链适合多层处理流程，比如登录校验、权限校验、参数校验、业务处理。

~~~text
请求 -> 登录检查 -> 权限检查 -> 参数检查 -> 业务处理
~~~

每一层只处理自己负责的部分，处理不了就交给下一层。

Web 框架里的中间件、网关过滤器、日志拦截器，都有责任链的影子。

## 模板方法：流程固定，步骤可变

模板方法适合“整体流程固定，但其中某些步骤由子类实现”的场景。

比如游戏对象更新流程固定为：处理状态、移动、碰撞、绘制。不同对象只重写其中某些步骤。

这样可以保证流程一致，同时允许局部差异。

## 组合：像树一样组织对象

组合模式适合树形结构，比如文件夹和文件、UI 容器和控件、菜单和子菜单。

它的特点是单个对象和对象集合使用相同接口。调用方不用关心自己拿到的是一个叶子节点，还是一整棵子树。

## 什么时候不要用模式

如果一段代码只有两个分支，而且未来很少变化，就没必要立刻上策略模式。

如果一个对象只是临时使用，也不需要单例。

如果一个项目还没稳定，过早抽象会让你维护一堆没必要的类。

判断是否要用模式，可以问三件事：

- 这个变化以后会反复出现吗？
- 现在代码是不是已经很难维护？
- 抽象之后调用方会更清楚吗？

只有答案比较明确时，再引入模式。

## 面试怎么讲

讲设计模式不要只背定义。更好的方式是先讲问题，再讲模式怎么解决，最后讲代价。

比如策略模式可以这样讲：当系统里有多种可替换算法时，如果全部写在 if-else 里，新增算法会频繁修改原代码。策略模式把算法拆成独立对象，调用方依赖抽象接口，新增策略时更符合开闭原则。但它也会增加类数量，简单场景可能过度设计。

## 总结

设计模式本质是代码结构经验。它们不是标准答案，而是帮助你控制变化的工具。真正重要的是判断边界：该抽象时抽象，该简单时简单。能把这个边界讲清楚，比背出二十三种模式更有价值。
