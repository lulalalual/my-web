## 通俗导读

操作系统可以理解成“管家”：它帮程序分配 CPU、内存、文件和设备。面试里问操作系统，通常不是让你背名词，而是看你能不能讲清程序运行时到底发生了什么。

```text
程序运行 -> 进入进程 -> 申请内存 -> 使用 CPU -> 发生 IO -> 可能切换线程
```

## 先看例子

```cpp
int main() {
    int x = 10;          // 栈上变量
    int* p = new int(20); // 堆上变量
    delete p;
}
```

这段代码背后就能引出很多操作系统问题：栈和堆怎么分配、虚拟内存怎么映射、进程地址空间怎么隔离。

## 阅读建议

先看内核态和用户态，再看虚拟内存、分页、进程线程和调度。遇到概念时，多问一句：这个机制解决了什么问题？

# 系统结构与内存管理

## 一、系统结构：内核（Kernel）

### 1. 内核的定义与作用

**内核**是操作系统的核心组件，运行在**最高特权级**，负责管理计算机的核心资源（CPU、内存、设备I/O、进程调度等），并为上层应用提供基础服务（如系统调用接口）。

- **核心功能**：进程管理、内存管理、文件系统、设备驱动、中断处理、安全机制。
    

### 2. 内核结构类型（常考对比）

根据功能模块的集成方式，内核可分为以下类型：

|**类型**|**结构特点**|**优点**|**缺点**|**代表系统**|
|:--|:--|:--|:--|:--|
|**宏内核**|所有核心功能（进程、内存、驱动等）集中在一个内核空间运行，模块间直接通信。|性能高（函数调用快）、结构紧凑。|稳定性差（一个模块崩溃可能导致内核崩溃）、扩展性弱。|Linux、Windows、macOS|
|**微内核**|仅保留最基础功能（进程通信、CPU调度、内存管理），其他功能（驱动、文件系统等）作为用户态服务运行。|稳定性高（服务崩溃不影响内核）、扩展性强。|性能低（频繁用户态-内核态切换）、设计复杂。|Minix、QNX、Mach（macOS部分）|
|**混合内核**|结合宏内核与微内核特点：核心功能在内核态，部分非核心功能（如驱动）可模块化加载到内核态或用户态。|平衡性能与稳定性。|设计复杂度介于两者之间。|Windows NT、macOS（XNU内核）|

### 3. 内核态 vs 用户态（CPU特权级）

CPU通过**特权级**（Ring Level）区分执行模式：

- **内核态（Ring 0）**：最高特权级，可执行所有指令（如访问硬件、修改页表），内核运行在此模式。
    
- **用户态（Ring 3）**：最低特权级，仅能执行受限指令（如算术运算、访问用户空间内存），应用程序运行在此模式。
    

## 二、内存管理：用户空间与内核空间

### 1. 为何划分用户空间与内核空间？

- **安全性**：防止用户程序直接访问硬件或修改内核数据（如页表、进程控制块），避免恶意破坏系统。
    
- **稳定性**：用户程序崩溃不会影响内核及其他程序（内核空间受硬件保护）。
    
- **资源管理**：内核统一管理物理内存、磁盘等稀缺资源，避免冲突。
    

### 2. 虚拟地址空间划分（32位 vs 64位系统）

现代操作系统采用**虚拟内存**技术，将进程的地址空间划分为用户空间和内核空间（逻辑隔离，物理上可能共享内存）。

#### （1）32位系统（经典布局）

- **总虚拟地址空间**：4GB（2³²字节）。
    
- **划分比例**：通常为 **用户空间3GB（0~3GB）+ 内核空间1GB（3GB~4GB）**（Linux默认，Windows可配置为2GB用户/2GB内核）。
    
- **用户空间**：每个进程独有，存放代码、数据、堆、栈、共享库等（如Linux的`.text`段、`.data`段、堆区、栈区）。
    
- **内核空间**：所有进程共享，存放内核代码、内核数据结构（如进程PCB、页表）、DMA缓冲区等（需注意：**内核空间映射到同一物理内存**，避免切换进程时重复加载内核）。
    

#### （2）64位系统（扩展布局）

- **总虚拟地址空间**：理论16EB（2⁶⁴字节），实际仅使用部分（如x86-64架构常用48位虚拟地址，支持256TB空间）。
    
- **划分比例**：用户空间与内核空间通常为 **128TB用户 + 128TB内核**（Linux）或更大，远大于实际需求，几乎不会耗尽。
    

### 3. 用户空间与内核空间的核心区别

|**维度**|**用户空间**|**内核空间**|
|:--|:--|:--|
|**所属权**|进程私有（每个进程独立映射）|所有进程共享（映射到同一物理内存区域）|
|**特权级**|Ring 3（低特权）|Ring 0（高特权）|
|**内存访问**|仅能访问自身虚拟地址空间（用户空间），越界触发段错误（SIGSEGV）。|可访问全部虚拟地址空间（用户+内核），可直接操作物理内存（通过MMU映射）。|
|**生命周期**|随进程创建/销毁而分配/释放。|系统启动时分配，一直驻留内存（除非主动卸载模块）。|
|**典型内容**|应用程序代码、堆、栈、共享库（.so/.dll）。|内核代码、进程PCB、页表、设备驱动、中断向量表。|

### 4. 用户态与内核态的切换（常考流程）

当应用程序需要内核服务（如读写文件、申请内存）时，需从**用户态切换到内核态**，核心流程如下：

#### 触发条件

- **系统调用**（主动请求，如`read()`/`write()`）；
    
- **硬件中断**（被动响应，如键盘输入、磁盘IO完成）；
    
- **异常**（被动处理，如除零错误、缺页中断）。
    

#### 切换步骤（以系统调用为例）

1. **用户态准备**：应用程序将系统调用号（如`sys_read`）和参数存入寄存器（如x86的`eax`、`ebx`）。
    
2. **触发软中断**：执行`int 0x80`（32位）或`syscall`指令（64位），CPU切换到内核态（Ring 0），并跳转到内核预设的中断入口（如`system_call`函数）。
    
3. **内核态处理**：
    
    - 保存用户态上下文（寄存器、程序计数器）到内核栈；
        
    - 根据系统调用号查询**系统调用表**（如`sys_call_table`），找到对应内核函数（如`sys_read`）；
        
    - 执行内核函数，完成实际操作（如读取磁盘数据到内核缓冲区）；
        
4. **返回用户态**：将结果存入寄存器，恢复用户态上下文，执行`iret`或`sysret`指令回到用户态。
    

### 5. 内存保护机制（如何隔离用户/内核空间）

- **MMU（内存管理单元）**：通过**页表**控制虚拟地址到物理地址的映射，页表中设置**权限位**（如`R/W/X`、用户/内核标志）。
    
    - 用户空间页表项标记为“用户态可访问”（U/S=1），内核空间页表项标记为“仅内核态可访问”（U/S=0）。
        
- **CPU特权检查**：访问内存时，CPU会验证当前特权级（CPL）与页表权限是否匹配，若用户态程序尝试访问内核空间，触发**页错误异常**（Page Fault）。
    

## 三、总结与高频面试问题

### 1. 核心总结

- **内核**是OS核心，分宏/微/混合内核，运行于内核态（Ring 0）。
    
- **用户/内核空间划分**是虚拟内存的一部分，32位系统通常3G/1G，64位系统更大，目的是安全与稳定。
    
- **用户态→内核态切换**通过系统调用/中断/异常触发，依赖软中断指令和MMU权限控制。
    

### 2. 高频面试问题

1. **为什么需要划分用户态和内核态？**
    
    答：安全（防用户程序破坏系统）、稳定（用户崩溃不影响内核）、资源管理（内核统一调度资源）。
    
2. **32位系统的虚拟地址空间布局是怎样的？**
    
    答：共4GB，通常用户空间0~3GB（进程私有），内核空间3GB~4GB（所有进程共享，映射到同一物理内存）。
    
3. **用户程序如何访问内核空间？**
    
    答：不能直接访问，需通过**系统调用**、**中断**或**异常**切换到内核态，由内核代为操作。
    
4. **宏内核和微内核的区别？**
    
    答：宏内核功能集中（如Linux），性能好但稳定性差；微内核仅保留核心功能（如Minix），服务运行在用户态，稳定性好但性能低。
    
5. **MMU在用户/内核空间隔离中的作用？**
    
    答：通过页表权限位（U/S位）标记用户/内核空间，用户态访问内核空间时触发页错误，由内核处理。
    

# 内存管理进阶

## 一、物理内存与虚拟内存

### 1. 物理内存（Physical Memory）

- **定义**：计算机硬件提供的实际内存（如DRAM），是程序运行时数据和指令的直接存储载体。
    
- **特点**：
    
    - 容量有限（如8GB、16GB），由硬件决定；
        
    - 直接被CPU访问，速度远快于磁盘（纳秒级 vs 毫秒级）；
        
    - 需通过**地址总线**寻址（如32位系统最大支持4GB物理内存）。
        

### 2. 虚拟内存（Virtual Memory）

- **定义**：操作系统通过**硬件（MMU）和软件（OS）配合**，为每个进程提供的**逻辑上连续、独立的地址空间**（通常比物理内存大），实现对物理内存的“扩展”。
    
- **核心作用**：
    
    - **解决物理内存不足**：将暂时不用的程序/数据存放在磁盘（交换空间），需要时调入内存；
        
    - **进程隔离**：每个进程拥有独立虚拟地址空间，避免相互干扰；
        
    - **简化编程**：程序员无需关心物理内存布局，直接使用连续虚拟地址。
        

### 3. 物理内存与虚拟内存的联系

- **映射关系**：通过**页表**（分页）或**段表**（分段）建立虚拟地址到物理地址的映射；
    
- **后备存储**：虚拟内存的“扩展部分”存储在磁盘的**交换空间**（Swap Space），当物理内存不足时，通过**缺页中断**将磁盘数据调入内存。
    

## 二、内存管理方式：分段与分页

### 1. 内存分段（Segmentation）

#### 核心思想

将程序按**逻辑功能**划分为不同“段”（如代码段、数据段、堆段、栈段），每段有独立虚拟地址空间，段内连续、段间不连续。

#### 实现方式

- **段表**：记录每个段的**基址**（物理内存起始地址）和**限长**（段大小）；
    
- **地址转换**：虚拟地址 = 段号 + 段内偏移，通过段号查段表，验证偏移 < 限长后，物理地址 = 基址 + 偏移。
    

#### 特点

|**优点**|**缺点**|
|:--|:--|
|符合程序逻辑结构（模块化）|段大小可变，易产生**外部碎片**（内存中零散空闲区无法利用）；|
|支持共享与保护（段级权限）|段表需存储基址+限长，空间开销大；|
|动态链接方便（按段加载）|段长需对齐，可能浪费空间。|

### 2. 内存分页（Paging）

#### 核心思想

将虚拟地址空间和物理内存均划分为**固定大小的“页”**（Page，如4KB），通过**页表**将虚拟页映射到物理页框（Frame）。

#### 实现方式

- **页**：虚拟地址空间的基本单位（如4KB/页）；
    
- **页框**：物理内存的基本单位（与页大小相同）；
    
- **页表**：记录虚拟页号到物理页框号的映射，含**有效位**（页是否在内存）、**保护位**（读/写/执行权限）等。
    

#### 地址转换流程

1. CPU生成虚拟地址，拆分为**虚拟页号（VPN）** 和**页内偏移（Offset）**；
    
2. 用VPN查页表，得到**物理页框号（PFN）**；
    
3. 物理地址 = PFN × 页大小 + Offset。
    

#### 特点

|**优点**|**缺点**|
|:--|:--|
|无外部碎片（页框大小固定）|存在**内部碎片**（页内未用满的空间，如4KB页存3KB数据，浪费1KB）；|
|页表统一，管理简单|页表可能过大（如32位系统4GB空间，4KB页，需1M个页表项，每项4B，共4MB/进程）；|
|支持按需调页（部分加载）|地址转换需访存（查页表），速度慢。|

### 3. 分段 vs 分页（面试高频对比）

|**维度**|**分段**|**分页**|
|:--|:--|:--|
|**划分单位**|逻辑单位（段，大小可变）|物理单位（页，大小固定，如4KB）|
|**碎片**|外部碎片（段间零散空闲区）|内部碎片（页内未用满空间）|
|**地址空间**|二维（段号+段内偏移）|一维（虚拟页号+页内偏移）|
|**共享与保护**|段级（按逻辑模块共享，如代码段）|页级（细粒度，可按页共享）|
|**实现复杂度**|较简单（段表结构简单）|较复杂（需多级页表、TLB等优化）|

## 三、多级页表与快表（TLB）

### 1. 多级页表（Multi-level Page Table）

#### 问题背景

单级页表在64位系统中空间爆炸：如64位虚拟地址（48位有效），4KB页，需 2^{36} 个页表项（每项8B，共 2^{39}B=512GB/进程），无法存储。

#### 核心思想

将页表**分级**，仅将**活跃页表项**驻留内存，非活跃部分存磁盘，减少空间占用。

#### 举例：二级页表（32位系统，4KB页）

- 虚拟地址32位，页大小4KB（12位偏移），虚拟页号20位；
    
- 将20位VPN分为**一级页号（10位）** 和**二级页号（10位）**；
    
- 一级页表（页目录）含 2^{10}=1024 项，每项指向一个二级页表（含1024项）；
    
- 仅当某二级页表有活跃页时，才为其分配内存。
    

#### 优点

- 节省空间：非活跃页表项不占内存；
    
- 支持稀疏地址空间（如大部分虚拟地址未使用）。
    

#### 缺点

- 地址转换需多次访存（如二级页表需访存2次+查页表），速度慢。
    

### 2. 快表（TLB，Translation Lookaside Buffer）

#### 核心作用

**页表的高速缓存**，存储近期访问的**虚拟页号→物理页框号**映射，加速地址转换（避免每次访存都查页表）。

#### 工作原理

1. CPU生成虚拟地址，提取VPN；
    
2. 先在TLB中查找VPN：
    
    - **命中（TLB Hit）**：直接获取PFN，物理地址=PFN+Offset，耗时1-2周期；
        
    - **未命中（TLB Miss）**：查多级页表获取PFN，更新TLB（替换旧条目），耗时10-100周期。
        

#### TLB特点

- **容量小**（如64-1024项）、**速度快**（SRAM实现，接近寄存器速度）；
    
- **全相联映射**（任意VPN可存于任意TLB项）；
    
- **替换策略**：LRU（最近最少使用）、FIFO、随机替换；
    
- **上下文切换时需清空**（不同进程页表不同，避免映射混淆）。
    

## 四、交换空间（Swap Space）与缺页中断

### 1. 交换空间（Swap Space）

#### 定义

磁盘上的一块专用区域，作为虚拟内存的“后备存储”，用于存放**暂时换出的内存页**。

#### 作用

- 当物理内存不足时，将不活跃的页从内存移到交换空间（“换出”）；
    
- 当访问被换出的页时，从交换空间调入内存（“换入”），触发**缺页中断**。
    

#### 类型

- **整体交换**：早期系统将整个进程换出（如Unix的“交换分区”）；
    
- **按需分页**（Demand Paging）：现代系统仅换出部分页（主流，如Linux的Swap分区/文件）。
    

### 2. 缺页中断（Page Fault）

#### 触发条件

当CPU访问的虚拟页**不在物理内存**（页表有效位=0）时，由MMU触发**缺页中断**（内中断的一种）。

#### 处理流程（面试重点）

1. **保护现场**：保存当前进程上下文（寄存器、PC）到内核栈；
    
2. **检查合法性**：判断访问的虚拟地址是否合法（如是否在进程地址空间内），若不合法则终止进程（段错误）；
    
3. **调页（Page In）**：
    
    - 若交换空间中有该页（被换出过），则从磁盘读入内存空闲页框；
        
    - 若从未加载过（如首次访问），则从磁盘文件（如可执行文件、共享库）读入；
        
4. **更新页表**：将新页框号填入页表，置有效位=1；
    
5. **恢复执行**：从被中断的指令继续执行（此时页已在内存）。
    

#### 特点

- 缺页中断是**同步中断**（执行指令时触发），优先级高于普通中断；
    
- 频繁缺页会导致“颠簸”（Thrashing），需通过页面置换算法缓解。
    

## 五、页面置换算法（Page Replacement Algorithms）

### 1. 核心问题

当物理内存满且需调入新页时，选择**哪个旧页换出**到交换空间，目标是最小化缺页率。

### 2. 常见算法（面试必背）

#### （1）先进先出（FIFO，First-In First-Out）

- **思想**：选择**最早进入内存**的页换出。
    
- **实现**：用队列维护页的进入顺序，队首为最早页。
    
- **优点**：实现简单（无需记录访问时间）。
    
- **缺点**：Belady异常（分配页框增多，缺页率反而上升）；未考虑页的使用频率，可能换出常用页。
    

#### （2）最佳置换（OPT，Optimal）

- **思想**：选择**未来最长时间不被访问**的页换出（理想算法，无法实际实现，用于对比）。
    
- **实现**：需预知未来访问序列（如编译器优化时模拟）。
    
- **优点**：缺页率最低。
    
- **缺点**：无法实际应用。
    

#### （3）最近最少使用（LRU，Least Recently Used）

- **思想**：选择**最近最久未访问**的页换出（基于“局部性原理”，常用页近期可能再用）。
    
- **实现**：
    
    - **硬件实现**：用移位寄存器或栈记录页的访问时间（成本高，现代CPU很少用）；
        
    - **软件近似**：
        
        - **Clock算法**（二次机会法）：用环形链表+访问位（1=最近访问过，0=未访问），换出时扫描，遇1则置0并跳过，遇0则换出；
            
        - **LRU-K**：记录页最近K次访问时间，K=1时近似LRU，K=2时更准确。
            
- **优点**：缺页率低，符合程序局部性。
    
- **缺点**：实现复杂（需硬件/软件配合）。
    

#### （4）时钟算法（Clock，近似LRU）

- **思想**：给每个页设**访问位**（1=已访问，0=未访问），用“时钟指针”循环扫描：
    
    - 若页的访问位=1，置0并跳过；
        
    - 若访问位=0，换出该页。
        
- **改进**：增加**修改位**（脏位），优先换出未修改页（无需写回磁盘）。
    

### 3. 算法对比表

|**算法**|**核心思想**|**优点**|**缺点**|**实际应用**|
|:--|:--|:--|:--|:--|
|FIFO|换出最早进入的页|实现简单|Belady异常，缺页率高|几乎不用|
|OPT|换出未来最久不用的页|缺页率最低|无法预知未来|理论对比基准|
|LRU|换出最近最久未用的页|缺页率低|硬件实现成本高|数据库缓存、CPU缓存|
|Clock|环形扫描，二次机会换出|近似LRU，成本低|不如LRU精确|Linux（近似LRU）、BSD|

## 六、总结与高频面试问题

### 1. 核心总结

- **虚拟内存**通过分页/分段扩展物理内存，依赖MMU和交换空间；
    
- **分页**无外部碎片但有内部碎片，**分段**反之；
    
- **多级页表**解决页表空间爆炸，**TLB**加速地址转换；
    
- **缺页中断**处理页不在内存的情况，**页面置换算法**决定换出哪个页（LRU最优但难实现，Clock常用）。
    

### 2. 高频面试问题

1. **为什么需要虚拟内存？**
    
    答：扩展物理内存、进程隔离、简化编程、支持按需调页。
    
2. **分页和分段的区别？**
    
    答：分页是物理单位（固定大小，无外部碎片），分段是逻辑单位（可变大小，无内部碎片）；分页一维地址，分段二维地址。
    
3. **多级页表解决了什么问题？缺点是什么？**
    
    答：解决单级页表空间过大的问题（仅存活跃页表项）；缺点是地址转换需多次访存，速度慢（需TLB优化）。
    
4. **TLB的作用是什么？命中率和什么有关？**
    
    答：缓存近期页表映射，加速地址转换；命中率与TLB容量、程序局部性有关（局部性好则命中率高）。
    
5. **缺页中断的处理流程？**
    
    答：保护现场→检查合法性→调页（读磁盘）→更新页表→恢复执行。
    
6. **LRU算法的实现难点？如何用Clock算法近似？**
    
    答：LRU需记录访问时间，硬件成本高；Clock算法用环形链表+访问位，扫描时给“第二次机会”，近似LRU效果。
    

**记忆口诀**：“物理内存硬件限，虚拟扩展靠磁盘；分段逻辑分页物，多级页表省空间；TLB缓存加速转，缺页中断调页忙；置换算法LRU优，Clock近似常用！”

# IO管理

## 一、IO基本概念与分类

### 1. IO操作的本质

IO（Input/Output）是计算机与外部设备（如磁盘、网卡、键盘）进行数据交换的过程。对CPU而言，IO操作是**低速操作**（相比CPU运算），因此需通过**内核**协调：用户进程发起IO请求后，内核负责与硬件交互，完成后通知进程。

### 2. IO模型的分类（常考）

根据**进程在IO操作中的状态**和**数据复制阶段的同步/异步**，IO模型可分为：

- **同步IO**：数据从内核缓冲区复制到用户缓冲区的过程需进程主动参与（阻塞/非阻塞、IO多路复用均属于同步IO）。
    
- **异步IO（AIO）**：内核完成数据复制后主动通知进程，进程无需参与复制（如Windows IOCP、Linux AIO）。
    

本文重点讲解**阻塞IO、非阻塞IO、IO多路复用**（均为同步IO）。

## 二、阻塞IO与非阻塞IO

### 1. 阻塞IO（Blocking IO）

#### 定义

进程发起IO请求后，**必须等待IO操作完成（数据从硬件→内核缓冲区→用户缓冲区）才能继续执行**，期间进程处于**阻塞状态**（让出CPU，加入等待队列）。

#### 流程（以“读文件”为例）

1. 用户进程调用`read(fd, buf, size)`，进入内核态；
    
2. 内核检查数据是否在内核缓冲区：
    
    - 若未就绪（如磁盘数据未读入内核），进程阻塞，加入等待队列；
        
    - 若已就绪，将数据从内核缓冲区复制到用户缓冲区；
        
3. 复制完成，进程解除阻塞，返回用户态，继续执行。
    

#### 特点

- **优点**：编程简单（无需处理IO就绪状态）；
    
- **缺点**：进程在IO期间无法执行其他任务，资源利用率低（如单线程处理多IO时，其他IO被阻塞）。
    

### 2. 非阻塞IO（Non-blocking IO）

#### 定义

进程发起IO请求后，**立即返回状态**（无论IO是否完成）：若IO未就绪，返回`EWOULDBLOCK`错误；若已就绪，则进行数据复制。进程需**主动轮询**检查IO是否完成。

#### 流程（以“读socket”为例）

1. 用户进程将socket设为非阻塞模式（`fcntl(fd, F_SETFL, O_NONBLOCK)`）；
    
2. 调用`read(fd, buf, size)`：
    
    - 若内核缓冲区无数据，立即返回-1（errno=EWOULDBLOCK），进程不阻塞，可执行其他任务；
        
    - 若数据就绪，复制数据到用户缓冲区，返回实际读取字节数；
        
3. 进程通过**轮询**（如循环调用`read`）检查IO是否完成。
    

#### 特点

- **优点**：进程在IO未就绪时可执行其他任务，避免阻塞；
    
- **缺点**：轮询消耗CPU资源（“忙等”），效率低下（尤其IO就绪频率低时）。
    

### 3. 阻塞IO vs 非阻塞IO（面试高频对比）

|**维度**|**阻塞IO**|**非阻塞IO**|
|:--|:--|:--|
|**IO未就绪时**|进程阻塞（让出CPU）|立即返回错误，进程继续执行（需轮询）|
|**CPU利用率**|低（阻塞时CPU闲置）|高（轮询消耗CPU，但可穿插其他任务）|
|**编程复杂度**|简单（无需处理轮询）|复杂（需主动轮询+错误处理）|
|**适用场景**|单IO操作、对延迟不敏感|多IO操作、需同时处理其他任务|

## 三、IO多路复用（IO Multiplexing）

### 1. 引入背景

阻塞IO无法同时处理多个IO（需多线程/多进程，资源消耗大），非阻塞IO的轮询又浪费CPU。**IO多路复用**允许**单个进程同时监控多个IO描述符**（如socket、文件），当某个描述符就绪（可读/可写/异常）时，通知进程处理，实现“用一个进程管理多个IO”。

### 2. 核心思想

- **事件驱动**：进程向内核注册“感兴趣的事件”（如“socket可读”），内核监控所有注册的描述符，当事件发生时，将就绪描述符通知进程；
    
- **同步IO**：数据从内核缓冲区复制到用户缓冲区的阶段仍需进程参与（区别于异步IO）。
    

### 3. 主流实现：select/poll/epoll

#### （1）select（POSIX标准，跨平台）

##### 实现原理

- **数据结构**：用`fd_set`（位图）存储要监控的描述符，每个bit对应一个描述符（如bit 3对应fd=3）；
    
- **流程**：
    
    1. 进程初始化`fd_set`（设置需监控的读/写/异常事件）；
        
    2. 调用`select(nfds, read_fds, write_fds, except_fds, timeout)`：
        
        - 内核遍历所有注册的描述符，检查是否就绪；
            
        - 若有就绪描述符，修改`fd_set`仅保留就绪项，返回就绪数量；
            
        - 若无就绪描述符且未超时，进程阻塞；
            
    3. 进程遍历`fd_set`，处理就绪描述符。
        

##### 缺点

- **描述符数量限制**：`fd_set`大小固定（通常1024，由`FD_SETSIZE`定义），无法监控大量描述符；
    
- **效率低**：每次调用需将`fd_set`从用户态拷贝到内核态，内核需遍历所有描述符（O(n)复杂度）；
    
- **重复初始化**：每次调用需重新设置`fd_set`（因内核会修改它）。
    

#### （2）poll（POSIX标准，改进select）

##### 实现原理

- **数据结构**：用`pollfd`数组替代`fd_set`，每个元素包含`fd`（描述符）、`events`（关注的事件）、`revents`（就绪的事件）；
    
- **流程**：
    
    1. 进程初始化`pollfd`数组（设置`fd`和`events`）；
        
    2. 调用`poll(fds, nfds, timeout)`：内核遍历数组检查就绪状态，修改`revents`；
        
    3. 进程遍历数组，处理`revents≠0`的描述符。
        

##### 改进与缺点

- **改进**：取消描述符数量限制（数组大小由用户定义），无需重复初始化`fd_set`；
    
- **缺点**：仍需遍历所有描述符（O(n)复杂度），拷贝数组开销仍存在。
    

#### （3）epoll（Linux特有，高性能）

##### 实现原理

- **核心组件**：
    
    - **红黑树**：存储所有注册的描述符（高效增删改查）；
        
    - **就绪链表**：存储就绪的描述符（内核检测到事件时加入）；
        
    - **回调机制**：当描述符就绪时，内核通过回调函数将其加入就绪链表。
        
- **流程**：
    
    1. 调用`epoll_create()`创建epoll实例（返回epoll句柄）；
        
    2. 调用`epoll_ctl(epfd, op, fd, event)`注册/修改/删除描述符及事件（如`EPOLLIN`表示可读）；
        
    3. 调用`epoll_wait(epfd, events, maxevents, timeout)`：
        
        - 若就绪链表非空，直接返回就绪描述符（O(1)复杂度）；
            
        - 若为空且未超时，进程阻塞；
            
    4. 进程处理`events`数组中的就绪描述符。
        

##### 两种触发模式

- **LT（水平触发，Level Triggered，默认）**：
    
    - 描述符就绪后，若数据未读完，下次`epoll_wait`仍会通知（反复通知，直到数据被处理）；
        
    - 优点：编程简单（不易遗漏事件）；缺点：可能重复触发，效率略低。
        
- **ET（边缘触发，Edge Triggered）**：
    
    - 仅在描述符状态变化时通知一次（如从“不可读”→“可读”时触发，数据未读完不再通知）；
        
    - 优点：减少通知次数，效率高；缺点：需一次性处理完数据（必须用非阻塞IO+循环读/写，避免阻塞）。
        

##### 优点

- **高效**：O(1)复杂度（仅返回就绪描述符），支持海量描述符（百万级）；
    
- **低开销**：无需重复拷贝描述符集合（通过`epoll_ctl`增删改）；
    
- **灵活**：支持LT/ET模式，适配不同场景。
    

### 4. select/poll/epoll对比（面试必背）

|**维度**|**select**|**poll**|**epoll**|
|:--|:--|:--|:--|
|**数据结构**|fd_set（位图，大小固定）|pollfd数组（无固定大小）|红黑树+就绪链表（动态扩容）|
|**描述符数量**|受限（默认1024）|无限制（数组大小自定义）|无限制（百万级）|
|**时间复杂度**|O(n)（遍历所有描述符）|O(n)（遍历所有描述符）|O(1)（仅返回就绪描述符）|
|**拷贝开销**|每次调用拷贝fd_set|每次调用拷贝pollfd数组|仅epoll_ctl时拷贝（增删改）|
|**触发模式**|仅LT|仅LT|LT/ET（边缘触发更高效）|
|**适用场景**|少量描述符（<1024）|中等描述符（无数量限制）|大量描述符（高并发服务器）|

## 四、IO多路复用的应用场景

- **高并发服务器**：如Web服务器（Nginx、Redis）用epoll监控大量客户端socket，单进程处理数万连接；
    
- **聊天服务器**：同时监控多个用户的消息收发；
    
- **实时数据处理**：如日志收集系统监控多个日志文件的写入事件。
    

## 五、总结与高频面试问题

### 1. 核心总结

- **阻塞IO**：IO未就绪时进程阻塞，简单但低效；
    
- **非阻塞IO**：IO未就绪时立即返回，需轮询，CPU利用率低；
    
- **IO多路复用**：单进程监控多个IO，通过select/poll/epoll实现，epoll性能最优（红黑树+就绪链表，O(1)复杂度）；
    
- **epoll ET模式**：需配合非阻塞IO，一次性处理完数据，避免遗漏事件。
    

### 2. 高频面试问题

1. **阻塞IO和非阻塞IO的区别？**
    
    答：阻塞IO在IO未就绪时进程阻塞；非阻塞IO立即返回，需轮询检查就绪状态。
    
2. **IO多路复用的作用是什么？**
    
    答：允许单个进程同时监控多个IO描述符，当某个描述符就绪时通知进程处理，避免多线程/多进程的资源消耗。
    
3. **select/poll/epoll的区别？**
    
    答：select用固定大小fd_set，O(n)复杂度；poll用数组无数量限制，仍O(n)；epoll用红黑树+就绪链表，O(1)复杂度，支持LT/ET模式，性能最优。
    
4. **epoll的LT和ET模式有何区别？**
    
    答：LT模式下，就绪事件会反复通知直到处理完；ET模式仅在状态变化时通知一次，需一次性处理完数据（配合非阻塞IO）。
    
5. **为什么epoll比select高效？**
    
    答：epoll用红黑树存储描述符（增删改查O(log n)），就绪链表直接返回就绪项（O(1)），无需遍历所有描述符，且减少用户态/内核态拷贝开销。
    

**记忆口诀**：“阻塞等待IO完，非阻塞轮询忙；多路复用管多IO，select poll epoll强；epoll红黑树+链表，LT重复ET一次，高并发服它最棒！”

# 进程管理

## 一、并行（Parallelism）与并发（Concurrency）

### 1. 核心定义

- **并行**：多个任务**在同一时刻同时执行**（依赖多核CPU或多处理器，物理上同时运行）。
    
    例：4核CPU同时运行4个进程，每个核执行一个进程。
    
- **并发**：多个任务**在同一时间段内交替执行**（单核CPU通过时间片切换实现“同时”假象）。
    
    例：单核CPU上，进程A执行10ms后切换到进程B执行10ms，宏观上“同时”运行。
    

### 2. 关键区别（面试高频）

|**维度**|**并行**|**并发**|
|:--|:--|:--|
|**执行方式**|同一时刻真正同时执行（物理并行）|同一时间段内交替执行（逻辑并行）|
|**硬件依赖**|必须多核/多处理器|单核/多核均可（单核通过切换实现）|
|**目标**|提高吞吐量（同时处理多任务）|提高资源利用率（减少CPU空闲）|
|**典型场景**|科学计算（多核并行计算）、大数据处理|服务器多客户端请求、GUI程序响应|

### 3. 关系总结

- **并行是并发的子集**：并行一定是并发（同时执行必然是交替执行的特例），并发不一定是并行（单核交替执行也是并发）。
    
- **现代OS同时支持**：多核CPU上，既有多任务的并发调度，也有单任务内的并行执行（如多线程并行）。
    

## 二、上下文切换（Context Switching）

### 1. 定义

当CPU从一个进程（或线程）切换到另一个进程时，**保存当前进程的状态（上下文）并加载新进程状态**的过程。

### 2. 上下文的内容

- **用户级上下文**：程序计数器（PC）、通用寄存器、浮点寄存器、用户栈指针。
    
- **内核级上下文**：内核栈指针、进程控制块（PCB，含进程状态、优先级、页表指针等）。
    
- **硬件上下文**：MMU寄存器（如TLB）、CPU状态字（PSW）。
    

### 3. 切换步骤（面试重点）

1. **保存旧进程上下文**：将当前寄存器、PC、PSW等存入内核栈（或PCB）。
    
2. **更新PCB状态**：将旧进程状态改为“就绪”或“阻塞”，加入对应队列。
    
3. **选择新进程**：调度器从就绪队列中选一个进程（如按调度算法）。
    
4. **加载新进程上下文**：从新进程的PCB中恢复寄存器、PC、页表等。
    
5. **切换地址空间**：更新MMU的页表基址寄存器（CR3），切换到新进程虚拟地址空间。
    
6. **执行新进程**：从新进程的PC处开始执行。
    

### 4. 切换开销（为何要减少切换？）

- **时间开销**：保存/加载上下文（约几微秒到毫秒级，依赖硬件和进程复杂度）。
    
- **CPU资源浪费**：切换过程中CPU不执行有效任务（“空转”）。
    
- **缓存失效**：新进程可能使用不同的内存页，导致TLB、L1/L2缓存失效（缓存抖动）。
    

## 三、进程状态（Process States）

### 1. 经典五状态模型（面试基础）

|**状态**|**定义**|**触发转换的事件**|
|:--|:--|:--|
|**创建（New）**|进程刚被创建，尚未进入就绪队列。|父进程调用fork()创建子进程。|
|**就绪（Ready）**|进程已获得除CPU外的所有资源，等待调度。|创建完成、IO完成、被抢占、阻塞解除。|
|**运行（Running）**|进程正在CPU上执行。|调度器选中该进程（就绪→运行）。|
|**阻塞（Blocked）**|进程因等待IO、信号量等资源而暂停执行。|发起IO请求、申请互斥锁失败、等待子进程结束。|
|**终止（Terminated）**|进程执行完毕或被终止（如exit()、kill()）。|正常退出、错误退出、被信号杀死。|

### 2. 扩展状态：挂起（Suspend）

为解决内存不足问题，将进程暂时换出到磁盘（交换空间），分为：

- **就绪挂起（Ready Suspended）**：进程在磁盘中，可被调入内存就绪队列。
    
- **阻塞挂起（Blocked Suspended）**：进程在磁盘中，等待的事件发生后转为就绪挂起。
    

### 3. 状态转换图（核心转换）

- **运行→就绪**：时间片用完（抢占式调度）、更高优先级进程就绪（抢占）。
    
- **运行→阻塞**：发起IO请求、等待信号量。
    
- **阻塞→就绪**：IO完成、信号量可用。
    
- **就绪→运行**：调度器选中（非抢占式）/抢占式调度触发。
    

## 四、特殊进程（孤儿进程、僵尸进程、守护进程）

### 1. 孤儿进程（Orphan Process）

- **定义**：父进程先于子进程终止，子进程成为“孤儿”。
    
- **处理**：被**init进程（PID=1）** 收养，由init进程回收资源（调用wait()）。
    
- **危害**：几乎无危害（系统自动接管）。
    

### 2. 僵尸进程（Zombie Process）

- **定义**：子进程终止后，父进程未调用wait()回收其PCB，导致子进程残留“僵尸”状态（Z状态）。
    
- **特征**：僵尸进程不占用内存/CPU，但占用PCB资源（PID、退出状态）。
    
- **危害**：大量僵尸进程会耗尽PID资源（系统无法创建新进程）。
    
- **处理**：
    
    - 父进程主动调用wait()/waitpid()回收；
        
    - 杀死父进程（僵尸进程转为孤儿，被init回收）；
        
    - 重启系统（极端情况）。
        

### 3. 守护进程（Daemon Process）

- **定义**：后台运行的“服务进程”，脱离终端控制，生命周期长（如系统服务）。
    
- **特征**：
    
    - 父进程为init（PID=1）；
        
    - 无控制终端（TTY=？）；
        
    - 通常以root权限运行（如sshd、cron）。
        
- **创建步骤**：
    
    1. fork()创建子进程，父进程exit()（子进程被init收养）；
        
    2. 子进程调用setsid()创建新会话（脱离终端）；
        
    3. 切换工作目录到根目录（避免占用挂载点）；
        
    4. 重定向标准输入/输出/错误到/dev/null；
        
    5. 设置文件掩码（umask(0)）。
        

## 五、进程通信方式（IPC，Inter-Process Communication）

### 1. 分类与核心方式（面试必背）

|**类型**|**具体方式**|**原理**|**特点**|**适用场景**|
|:--|:--|:--|:--|:--|
|**共享存储**|共享内存（Shared Memory）|多个进程映射同一块物理内存，直接读写（需同步机制如信号量）。|最快（无数据拷贝），需手动同步（易死锁）。|大数据量交换（如数据库缓存）|
|**消息传递**|消息队列（Message Queue）|内核维护消息链表，进程通过send/receive发送/接收结构化消息（含类型、数据）。|支持多对多通信，内核缓冲（不怕丢失），但拷贝开销大。|异步通信（如日志收集）|
||直接通信（Direct Communication）|进程指定目标进程ID，直接发送消息（如RPC）。|简单，但耦合度高（需知道对方ID）。|一对一通信|
||间接通信（Indirect Communication）|通过“信箱”（内核对象）中转消息（如邮箱）。|松耦合（信箱独立存在），支持多对多。|多进程协作|
|**管道（Pipe）**|匿名管道（Anonymous Pipe）|半双工通信（单向），仅父子/兄弟进程通过fork()继承文件描述符。|基于文件系统，字节流传输，容量有限（如4KB）。|Shell命令（如ls|
||命名管道（Named Pipe/FIFO）|有文件名，任意进程通过open()打开（FIFO文件）。|支持任意进程通信，半双工，持久化（文件系统中可见）。|无亲缘关系进程通信|
|**信号量（Semaphore）**|计数信号量/二进制信号量|内核维护计数器，通过P（wait，减1）/V（signal，加1）操作同步临界资源。|低级通信（仅同步，无数据传输），用于互斥（二进制信号量）和资源计数。|生产者-消费者模型、互斥锁|
|**套接字（Socket）**|网络套接字（Network Socket）|基于TCP/UDP协议，跨主机通信（IP+端口标识进程）。|最通用（本地/远程均可），支持全双工，开销大（网络协议栈）。|分布式系统、网络通信|

### 2. 高级通信 vs 低级通信

- **低级通信**：信号量、管道（仅传递简单信号或字节流，效率低）。
    
- **高级通信**：共享内存、消息队列、套接字（传递结构化数据，效率高，功能丰富）。
    

## 六、进程调度算法（Process Scheduling Algorithms）

### 1. 调度目标

- **公平性**：每个进程获得合理的CPU时间。
    
- **高效性**：最大化CPU利用率、吞吐量（单位时间完成任务数）。
    
- **低延迟**：减少平均等待时间、响应时间（交互式系统）。
    
- **避免饥饿**：长期等待的进程最终能获得执行。
    

### 2. 经典调度算法（面试必背）

#### （1）先来先服务（FCFS，First-Come First-Served）

- **思想**：按进程到达就绪队列的顺序调度（非抢占式）。
    
- **优点**：实现简单（队列即可），公平。
    
- **缺点**：长作业“霸占”CPU，短作业等待时间长（平均等待时间高），对IO密集型进程不友好。
    
- **适用场景**：批处理系统（无交互需求）。
    

#### （2）短作业优先（SJF，Shortest Job First）

- **思想**：优先调度**估计运行时间最短**的进程（非抢占式为SJF，抢占式为SRTF，Shortest Remaining Time First）。
    
- **优点**：平均等待时间、周转时间最短（理论最优）。
    
- **缺点**：需预知运行时间（难实现），长作业可能饥饿（永远等不到执行）。
    
- **适用场景**：批处理系统（已知作业运行时间）。
    

#### （3）优先级调度（Priority Scheduling）

- **思想**：每个进程分配优先级（数值越小/越大优先级越高），优先调度高优先级进程。
    
- **分类**：
    
    - **非抢占式**：高优先级进程就绪时，等当前进程结束再调度。
        
    - **抢占式**：高优先级进程就绪时，立即抢占CPU。
        
- **问题**：低优先级进程饥饿（解决方案：老化（Aging），随时间增加优先级）。
    
- **适用场景**：实时系统（硬实时/软实时）、多用户系统（用户等级对应优先级）。
    

#### （4）时间片轮转（RR，Round Robin）

- **思想**：就绪队列按FCFS排队，每个进程分配一个**时间片**（如10-100ms），时间片用完则放回队尾（抢占式）。
    
- **优点**：公平（每个进程轮流执行），响应时间短（交互式友好）。
    
- **缺点**：时间片过大→退化为FCFS；时间片过小→切换开销占比高（吞吐量下降）。
    
- **适用场景**：分时系统（如Linux默认调度器CFS的基础）。
    

#### （5）多级反馈队列（MFQ，Multilevel Feedback Queue）

- **思想**：结合多种算法，设置多个优先级队列（Q1~Qn，Q1优先级最高），每个队列时间片递增（Q1时间片小，Qn大）。
    
- **规则**：
    
    1. 新进程进入Q1，按RR调度；
        
    2. 时间片用完未结束→降级到Q2（时间片增大）；
        
    3. 高优先级队列空时，才调度低优先级队列；
        
    4. 低优先级进程运行久后，可升级（避免饥饿）。
        
- **优点**：兼顾短作业（Q1快速执行）、长作业（Qn大时间片）、IO密集型（IO后升级到高优先级），综合性能最优。
    
- **适用场景**：通用OS（如Unix、Linux的CFS调度器借鉴其思想）。
    

### 3. 算法对比表

|**算法**|**类型**|**核心思想**|**优点**|**缺点**|**适用场景**|
|:--|:--|:--|:--|:--|:--|
|FCFS|非抢占式|先到先服务|简单、公平|长作业等待久，平均等待时间长|批处理系统|
|SJF/SRTF|非抢占/抢占|短作业优先|平均等待时间最短|需预知时间，长作业饥饿|批处理（已知时间）|
|优先级调度|非抢占/抢占|高优先级优先|灵活（按需求设优先级）|低优先级饥饿（需老化）|实时系统、多用户系统|
|RR|抢占式|时间片轮转|公平、响应快|时间片选择敏感（开销与延迟平衡）|分时系统|
|MFQ|抢占式|多级队列+反馈（升级/降级）|综合最优（兼顾长短作业）|实现复杂|通用OS（Unix、Linux）|

## 七、总结与高频面试问题

### 1. 核心总结

- **并行vs并发**：并行是物理同时（多核），并发是逻辑同时（单核切换）。
    
- **上下文切换**：保存/加载进程状态，开销大（时间+CPU+缓存失效）。
    
- **进程状态**：五状态（创建、就绪、运行、阻塞、终止），挂起状态应对内存不足。
    
- **特殊进程**：孤儿（父亡子存，init收养）、僵尸（子亡父未收，占PCB）、守护（后台服务，脱离终端）。
    
- **IPC方式**：共享内存（最快）、消息队列（异步）、管道（亲缘进程）、信号量（同步）、套接字（跨主机）。
    
- **调度算法**：FCFS简单、SJF最优但难实现、RR公平、MFQ综合最优（通用OS首选）。
    

### 2. 高频面试问题

1. **并行和并发的区别？**
    
    答：并行是同一时刻同时执行（多核），并发是同一时间段交替执行（单核切换）；并行是并发的子集。
    
2. **上下文切换的开销包括哪些？**
    
    答：保存/加载上下文的时间、CPU资源浪费、缓存失效（TLB/缓存抖动）。
    
3. **僵尸进程和孤儿进程的区别？**
    
    答：僵尸进程是子进程终止后父进程未回收（占PCB），孤儿进程是父进程先终止（被init收养，无害）。
    
4. **进程通信中，共享内存和消息队列的区别？**
    
    答：共享内存直接读写（无拷贝，需同步），消息队列通过内核缓冲（有拷贝，异步安全）。
    
5. **为什么多级反馈队列（MFQ）是通用OS的首选？**
    
    答：兼顾短作业（高优先级队列快速执行）、长作业（低优先级大时间片）、IO密集型（IO后升级），综合性能最优。
    

# 线程管理

## 一、线程（Thread）与进程（Process）的区别

### 1. 核心定义

- **进程**：资源分配的基本单位（拥有独立地址空间、文件描述符、PCB等），是程序的一次执行过程。
    
- **线程**：CPU调度的基本单位（轻量级进程），是进程内的执行流，共享进程资源（地址空间、文件描述符等）。
    

### 2. 关键区别（面试高频对比）

|**维度**|**进程**|**线程**|
|:--|:--|:--|
|**资源拥有**|拥有独立资源（地址空间、PCB、文件描述符）|共享进程资源（仅拥有私有栈、寄存器和TCB）|
|**调度单位**|传统OS中是调度单位（现代OS以线程为调度单位）|OS调度的基本单位（轻量级，切换开销小）|
|**并发性**|进程间并发（需切换地址空间）|线程间并发（同一进程内，共享资源，切换快）|
|**系统开销**|创建/销毁、切换开销大（需分配/释放资源）|创建/销毁、切换开销小（仅切换线程上下文）|
|**通信方式**|需IPC（共享内存、消息队列等）|直接读写共享内存（需同步机制避免冲突）|
|**健壮性**|进程崩溃不影响其他进程（隔离性好）|线程崩溃可能导致整个进程崩溃（共享资源）|

### 3. 关系总结

- **线程是进程的子集**：一个进程可包含多个线程（多线程），线程是进程内的执行单元。
    
- **现代OS的趋势**：以线程为调度单位（如Linux的轻量级进程LWP），进程作为资源容器。
    

## 二、线程上下文切换（Thread Context Switching）

### 1. 定义

CPU从一个线程切换到另一个线程时，**保存当前线程的状态（上下文）并加载新线程状态**的过程。

### 2. 上下文内容（比进程切换简单）

- **线程私有数据**：程序计数器（PC）、通用寄存器、栈指针（SP）、状态寄存器（PSW）。
    
- **线程控制块（TCB）**：存储线程ID、优先级、状态（就绪/运行/阻塞）、栈信息等。
    

### 3. 切换步骤（对比进程切换）

1. **保存旧线程上下文**：将寄存器、PC、SP存入TCB。
    
2. **更新TCB状态**：旧线程状态改为“就绪”或“阻塞”，加入就绪/阻塞队列。
    
3. **选择新线程**：调度器从就绪队列中选一个线程（如按优先级、时间片）。
    
4. **加载新线程上下文**：从新线程TCB恢复寄存器、PC、SP。
    
5. **执行新线程**：从新线程的PC处开始执行。
    

### 4. 与进程切换的核心差异

- **无需切换地址空间**：线程共享进程地址空间，无需更新MMU页表（CR3寄存器不变）。
    
- **开销更小**：仅切换线程私有数据（寄存器、栈），不涉及资源分配/释放，速度比进程切换快10-100倍。
    

## 三、线程的实现方式

### 1. 用户级线程（ULT，User-Level Thread）

#### 核心思想

线程管理（创建、切换、调度）由**用户空间线程库**（如POSIX Pthreads）实现，内核不知晓线程存在，仅将进程视为单线程。

#### 实现细节

- **TCB存储在用户空间**：线程库维护TCB表，调度算法自定义（如FCFS、优先级）。
    
- **切换无需内核介入**：通过函数调用实现线程切换（如`swapcontext`）。
    

#### 优点

- **切换速度快**：用户态完成，无需陷入内核（无模式切换开销）。
    
- **可定制调度**：用户可根据需求设计调度算法（如实时系统）。
    
- **跨平台**：不依赖内核支持（如嵌入式系统）。
    

#### 缺点

- **阻塞问题**：一个线程阻塞（如IO）会导致整个进程阻塞（内核仅调度进程）。
    
- **无法利用多核**：内核视进程为单线程，无法在多核CPU上并行执行多个线程。
    

#### 代表系统

早期UNIX（如Solaris GreenThreads）、GNU Portable Threads。

### 2. 内核级线程（KLT，Kernel-Level Thread）

#### 核心思想

线程管理由**内核**直接实现，内核维护TCB，负责线程的创建、切换和调度（如Linux的`task_struct`）。

#### 实现细节

- **TCB存储在内核空间**：内核通过系统调用（如`clone`）创建线程。
    
- **切换需内核介入**：通过软中断陷入内核，由内核调度器完成切换。
    

#### 优点

- **充分利用多核**：内核可将多个线程分配到不同CPU核心并行执行。
    
- **阻塞不影响进程**：一个线程阻塞（如IO），内核可调度同进程的其他线程执行。
    

#### 缺点

- **切换开销大**：需陷入内核（模式切换），速度慢于ULT。
    
- **内核负担重**：线程数量过多时，内核调度开销增大（如大量线程竞争CPU）。
    

#### 代表系统

Windows NT、Linux（NPTL线程库）、Solaris。

### 3. 混合实现（M:N模型，Hybrid Model）

#### 核心思想

结合ULT和KLT：**M个用户级线程映射到N个内核级线程**（M≥N），由用户线程库管理ULT，内核管理KLT。

#### 实现细节

- **ULT绑定KLT**：多个ULT可共享一个KLT（ULT阻塞时，KLT可调度其他ULT）。
    
- **动态调整映射**：根据负载动态增减KLT数量（如Solaris的LWP轻量级进程）。
    

#### 优点

- **兼顾速度与并行性**：ULT切换快，KLT可利用多核。
    
- **灵活性高**：用户库和内核协同调度，适应不同场景。
    

#### 缺点

- **实现复杂**：需协调用户库与内核的映射关系（如线程阻塞时的映射迁移）。
    

#### 代表系统

Solaris（LWP）、FreeBSD、Go语言的GMP调度模型（Goroutine-Machine-Processor）。

## 四、线程间同步（Thread Synchronization）

### 1. 同步的必要性

多线程共享进程资源（如全局变量、堆内存），若无序访问会导致**竞态条件**（Race Condition），需用同步机制保证**互斥**（Mutual Exclusion）和**协作**（Cooperation）。

### 2. 核心同步机制（面试必背）

#### （1）互斥锁（Mutex，Mutual Exclusion Lock）

- **原理**：同一时刻仅允许一个线程持有锁，其他线程需阻塞等待（“加锁-临界区-解锁”）。
    
- **API**：`pthread_mutex_lock()`（加锁）、`pthread_mutex_unlock()`（解锁）、`pthread_mutex_trylock()`（非阻塞尝试加锁）。
    
- **适用场景**：保护临界资源（如全局变量、共享数据结构）。
    

#### （2）条件变量（Condition Variable）

- **原理**：线程可在条件不满足时阻塞等待，直到其他线程发出“条件成立”的信号（需配合互斥锁使用）。
    
- **API**：`pthread_cond_wait()`（等待条件）、`pthread_cond_signal()`（唤醒一个等待线程）、`pthread_cond_broadcast()`（唤醒所有等待线程）。
    
- **适用场景**：生产者-消费者模型（消费者等待缓冲区非空，生产者通知）。
    

#### （3）信号量（Semaphore）

- **原理**：内核维护一个计数器（非负整数），通过`P操作`（wait，计数器减1，为0则阻塞）和`V操作`（signal，计数器加1，唤醒阻塞线程）控制资源访问。
    
- **分类**：
    
    - **二进制信号量**（0/1）：等价于互斥锁；
        
    - **计数信号量**（≥0）：控制多个同类资源（如缓冲区大小）。
        
- **适用场景**：限制并发访问资源的线程数（如连接池）。
    

#### （4）读写锁（Read-Write Lock）

- **原理**：区分“读操作”（共享）和“写操作”（独占）：允许多个线程同时读，仅允许一个线程写（读写互斥、写写互斥）。
    
- **API**：`pthread_rwlock_rdlock()`（读锁）、`pthread_rwlock_wrlock()`（写锁）、`pthread_rwlock_unlock()`（解锁）。
    
- **适用场景**：读多写少的场景（如缓存系统、配置文件读取）。
    

#### （5）原子操作（Atomic Operations）

- **原理**：不可中断的单步操作（如`i++`拆分为“读-改-写”，原子操作保证三步连续执行），无需锁即可保证线程安全。
    
- **实现**：硬件支持（如x86的`LOCK`前缀指令）、编译器内置函数（如GCC的`__atomic_add_fetch`）。
    
- **适用场景**：简单变量的增减（如计数器、标志位）。
    

#### （6）屏障（Barrier）

- **原理**：多个线程到达屏障点后等待，直到所有线程都到达才继续执行（“同步点”）。
    
- **API**：`pthread_barrier_init()`（初始化屏障）、`pthread_barrier_wait()`（等待屏障）。
    
- **适用场景**：多线程分阶段任务（如并行计算中“所有线程完成阶段1后再进入阶段2”）。
    

## 五、死锁（Deadlock）

### 1. 定义

多个线程因争夺资源而互相等待，若无外力干涉，它们都无法继续执行的状态（“循环等待”）。

### 2. 死锁的四个必要条件（缺一不可）

1. **互斥条件**：资源只能被一个线程独占（如互斥锁）。
    
2. **占有并等待**：线程持有至少一个资源，同时等待获取其他线程持有的资源。
    
3. **不可剥夺条件**：线程已获得的资源不能被其他线程强行夺取（只能主动释放）。
    
4. **循环等待条件**：多个线程形成环形等待链（如T1等待T2的资源，T2等待T3的资源，T3等待T1的资源）。
    

### 3. 死锁的处理方法

#### （1）预防死锁（破坏必要条件）

- **破坏互斥**：将独占资源改为共享资源（如只读文件无需互斥），但多数资源无法共享（如打印机）。
    
- **破坏占有并等待**：线程申请资源时必须一次性申请所有所需资源（“预先分配”），否则不执行（易导致资源利用率低）。
    
- **破坏不可剥夺**：允许线程抢占其他线程的资源（需保存被抢占线程的状态，复杂度高）。
    
- **破坏循环等待**：对资源编号，线程按顺序申请资源（如先申请编号小的资源，再申请大的），避免环形等待。
    

#### （2）避免死锁（动态判断是否安全）

- **银行家算法**（Banker's Algorithm）：线程申请资源时，内核模拟分配后判断系统是否处于“安全状态”（所有线程可顺利完成），若安全则分配，否则拒绝。
    
- **缺点**：需预知线程最大资源需求，实现复杂（实际系统少用）。
    

#### （3）检测与恢复

- **检测**：定期运行算法检查是否存在循环等待链（如资源分配图化简）。
    
- **恢复**：
    
    - **终止线程**：杀死一个或多个死锁线程（简单但可能丢失数据）；
        
    - **剥夺资源**：从死锁线程中剥夺资源分配给其他线程（需回滚线程状态）。
        

### 4. 经典例子：哲学家进餐问题

5个哲学家围坐圆桌，每人左右各有一根筷子（资源），需同时拿起两根筷子才能进餐。若所有哲学家同时拿起左边筷子，会导致循环等待（死锁）。

**解法**：限制同时拿筷子的哲学家数量（如最多4人）、奇数号先左后右、偶数号先右后左、引入服务员协调。

## 六、活锁（Livelock）与饥饿（Starvation）

### 1. 活锁（Livelock）

#### 定义

线程未被阻塞，但因不断重试无效操作而无法进展（类似“原地踏步”）。

#### 例子

两人走廊相遇，同时侧身让路，结果反复避让（都向左让，又都向右让，永远无法通过）。

#### 解决方法

- 引入随机退避（如线程失败后随机等待一段时间再重试）；
    
- 改变重试策略（如一方固定让路，另一方直行）。
    

### 2. 饥饿（Starvation）

#### 定义

线程因长期得不到所需资源（如CPU时间、锁）而无法执行（“饿死”）。

#### 原因

- **优先级调度**：低优先级线程长期被高优先级线程抢占；
    
- **资源分配不均**：某些线程总是抢先获得资源（如短作业优先导致长作业饥饿）；
    
- **同步机制缺陷**：如条件变量唤醒顺序不公平（总是唤醒后到的线程）。
    

#### 解决方法

- **老化机制（Aging）**：随时间增加低优先级线程的优先级（如每等待1秒优先级+1）；
    
- **公平调度**：使用公平锁（如按申请顺序分配锁）、轮转调度（RR）。
    

## 七、总结与高频面试问题

### 1. 核心总结

- **线程vs进程**：线程是轻量级执行流，共享进程资源，切换开销小；进程是资源容器，隔离性好。
    
- **线程实现**：ULT（用户库管理，快但阻塞影响进程）、KLT（内核管理，并行好但切换慢）、M:N（混合，平衡两者）。
    
- **同步机制**：互斥锁（独占）、条件变量（等待条件）、信号量（计数）、读写锁（读共享写独占）、原子操作（无锁简单操作）。
    
- **死锁**：需四个必要条件，处理方法是预防（破环条件）、避免（银行家算法）、检测恢复（终止/剥夺资源）。
    
- **活锁与饥饿**：活锁是不断重试不进展（随机退避解决），饥饿是长期得不到资源（老化机制解决）。
    

### 2. 高频面试问题

1. **线程和进程的区别？**
    
    答：线程共享进程资源（地址空间等），切换开销小，是调度单位；进程拥有独立资源，切换开销大，是资源容器。
    
2. **用户级线程和内核级线程的区别？**
    
    答：ULT由用户库管理（切换快、无法利用多核），KLT由内核管理（切换慢、可利用多核）。
    
3. **死锁的四个必要条件？如何破坏？**
    
    答：互斥、占有并等待、不可剥夺、循环等待。破坏方法：互斥（资源共享）、占有并等待（预先分配）、不可剥夺（允许抢占）、循环等待（按顺序申请资源）。
    
4. **线程同步有哪些方式？各适用于什么场景？**
    
    答：互斥锁（临界资源）、条件变量（等待条件）、信号量（计数资源）、读写锁（读多写少）、原子操作（简单变量）、屏障（多线程同步点）。
    
5. **活锁和饥饿的区别？**
    
    答：活锁是线程未被阻塞但不断重试无效操作（原地踏步）；饥饿是线程长期得不到资源（饿死），原因是优先级或分配不均。
    

# 基础知识与文件链接

## 一、操作系统基础知识

### 1. 操作系统的定义与核心作用

**操作系统（Operating System, OS）** 是管理计算机**硬件资源**（CPU、内存、磁盘、设备）和**软件资源**（程序、数据）的**系统软件**，为用户和应用程序提供**接口**，并负责协调资源分配与任务调度。

#### 核心作用（面试必背）

- **资源管理者**：分配CPU时间、内存空间、磁盘IO、设备使用权；
    
- **用户接口**：提供命令行（CLI）、图形界面（GUI）、系统调用（API）；
    
- **程序执行环境**：加载程序、管理进程/线程、处理异常与中断；
    
- **安全与保护**：隔离用户进程、控制资源访问权限（如文件读写权限）。
    

### 2. 操作系统的主要特征

|**特征**|**定义**|**例子**|
|:--|:--|:--|
|**并发**|多个任务在同一时间段内交替执行（单核）或同时执行（多核），宏观上“同时”。|单核CPU上浏览器与音乐播放器同时运行。|
|**共享**|资源被多个任务共同使用（互斥共享：如打印机；同时共享：如内存）。|多个进程共享物理内存、磁盘文件。|
|**虚拟**|将物理资源抽象为逻辑上的“虚拟资源”，扩大资源容量或数量。|虚拟内存（扩展物理内存）、虚拟CPU（时分复用）。|
|**异步**|任务执行速度不确定（受资源分配影响），但结果可预测（需通过同步机制保证）。|多进程并发执行时，完成顺序不确定。|

### 3. 操作系统的发展历程（简史）

|**阶段**|**时间**|**核心特点**|**代表系统**|
|:--|:--|:--|:--|
|**手工操作**|1940s-1950s|用户直接操作硬件（插线、拨开关），无OS，效率极低。|-|
|**批处理系统**|1950s-1960s|作业成批提交，由监督程序自动调度（单道批处理：一次一个作业；多道批处理：内存同时存多个作业）。|IBM OS/360|
|**分时系统**|1960s-1970s|多用户通过终端共享主机，按时间片轮转调度（响应快，交互性好）。|UNIX（早期）、CTSS|
|**实时系统**|1960s至今|严格按时序响应外部事件（硬实时：截止时间必须满足，如导弹控制；软实时：允许偶尔超时，如视频播放）。|VxWorks、RT-Linux|
|**现代OS**|1980s至今|支持多核、网络、分布式、虚拟化（如云计算），融合分时与实时特性。|Linux、Windows、macOS、Android|

### 4. 操作系统的体系结构（常考分类）

|**结构类型**|**核心思想**|**优点**|**缺点**|**代表系统**|
|:--|:--|:--|:--|:--|
|**单体结构**|所有功能模块（进程、内存、文件）集中在内核，直接调用（无层次）。|性能高（函数调用快）、结构紧凑。|稳定性差（模块耦合高，一处崩溃全崩）、扩展性弱。|早期UNIX、MS-DOS|
|**分层结构**|按功能分层（如硬件层、内存层、进程层、用户层），上层依赖下层接口。|模块化、易调试（逐层验证）。|效率低（层间调用开销大）。|THE系统（Dijkstra设计）|
|**微内核结构**|仅保留核心功能（进程通信、CPU调度、内存管理），其他功能（驱动、文件系统）作为用户态服务。|稳定性高（服务崩溃不影响内核）、扩展性强。|性能低（频繁用户态-内核态切换）。|Minix、QNX、Mach（macOS基础）|
|**客户-服务器**|基于微内核，将OS功能划分为多个“服务器进程”（如文件服务器、进程服务器），用户进程通过消息通信调用服务。|高度模块化、支持分布式。|实现复杂、消息通信开销大。|Mach、Windows NT|

## 二、软连接与硬连接（文件链接）

### 1. 前置知识：inode与目录项

在讲解链接前，需明确文件系统的核心概念：

- **inode（索引节点）**：存储文件的**元数据**（权限、大小、创建时间、数据块指针等），**不存储文件名**。每个文件有唯一inode号（同一文件系统内唯一）。
    
- **目录项（dentry）**：存储**文件名**与**inode号**的映射关系（如`/home/user/file.txt`中，“file.txt”是文件名，对应一个inode号）。
    
- **数据块（Data Block）**：存储文件的实际内容，由inode中的指针指向。
    

### 2. 硬连接（Hard Link）

#### 定义

**硬连接**是通过**多个目录项指向同一个inode**，使多个文件名关联同一文件数据的链接方式。硬连接与原文件**地位平等**，无“源文件”与“链接文件”之分。

#### 实现原理

- 创建一个硬连接时，文件系统会在目标目录下新增一个目录项，其inode号与原文件**完全相同**，并增加原inode的**链接计数（link count）**（初始为1，每新增一个硬连接+1）。
    
- 删除文件时，仅减少链接计数，当计数减为0时，才释放inode和数据块。
    

#### 特点（面试重点）

- **共享inode**：硬连接与原文件共享同一inode，因此**修改任一文件，另一文件同步变化**（本质是同一文件的不同名字）。
    
- **不能跨文件系统**：不同文件系统的inode号独立，无法指向其他文件系统的inode。
    
- **不能链接目录**：避免创建循环链接（如`ln /home /home/user/link`会导致`/home/user/link/home/...`无限循环），增加文件系统复杂度。
    
- **删除无影响**：删除硬连接或原文件，仅减少链接计数，数据不丢失（除非计数为0）。
    

#### 创建与查看（Linux示例）

```bash
# 创建硬连接（file2是file1的硬连接）
ln file1 file2  

# 查看inode号（两者相同）
ls -i file1 file2  # 输出：12345 file1  12345 file2  

# 查看链接计数（第三列数字）
ls -l file1  # 输出：-rw-r--r-- 2 user user 0 Jan 1 00:00 file1（链接计数为2）
```

### 3. 软连接（Symbolic Link，符号链接）

#### 定义

**软连接**是一个**特殊的文件**，其内容是**目标文件的路径**（类似Windows“快捷方式”）。软连接独立于原文件，有自己的inode和数据块（存储路径字符串）。

#### 实现原理

- 创建软连接时，文件系统会新建一个inode（类型标记为“符号链接”），数据块中存储**目标文件的绝对/相对路径**。
    
- 访问软连接时，系统会自动解析路径，找到目标文件（若目标不存在，软连接称为“断链”，访问时报错）。
    

#### 特点（面试重点）

- **独立inode**：软连接有自己的inode和数据块，与原文件无关。
    
- **可跨文件系统**：路径字符串可指向其他文件系统的文件（如`ln -s /mnt/disk2/file1 /home/user/link`）。
    
- **可链接目录**：支持链接目录（如`ln -s /usr/local /home/user/local_link`）。
    
- **目标删除后失效**：若原文件被删除，软连接变为“断链”（访问时提示“No such file or directory”）。
    
- **权限无关**：软连接的权限位通常显示为`lrwxrwxrwx`（实际权限由目标文件决定）。
    

#### 创建与查看（Linux示例）

```bash
# 创建软连接（link2指向file1）
ln -s file1 link2  

# 查看inode号（两者不同）
ls -i file1 link2  # 输出：12345 file1  67890 link2  

# 查看链接内容（存储目标路径）
readlink link2  # 输出：file1  

# 目标删除后，软连接失效
rm file1  
cat link2  # 报错：No such file or directory
```

### 4. 软连接 vs 硬连接（面试必背对比）

|**维度**|**硬连接**|**软连接**|
|:--|:--|:--|
|**本质**|多个目录项指向同一inode（同一文件）|独立文件，存储目标文件路径（快捷方式）|
|**inode**|与原文件共享inode（inode号相同）|有独立inode（inode号不同）|
|**跨文件系统**|不支持（inode号仅在本文件系统唯一）|支持（路径可指向其他文件系统）|
|**链接目录**|不支持（防止循环链接）|支持（可链接目录）|
|**目标删除影响**|无影响（链接计数-1，数据不删）|软连接失效（断链）|
|**文件大小**|与原文件相同（共享数据块）|很小（仅存储路径字符串，如`file1`占5B）|
|**创建命令**|`ln 源文件 目标文件`|`ln -s 源文件 目标文件`|

### 5. 应用场景

|**类型**|**适用场景**|
|:--|:--|
|**硬连接**|1. 防止误删重要文件（多目录项指向同一文件，删除一个不影响）；  <br>2. 备份文件（不占用额外空间，仅增加链接计数）；  <br>3. 同一文件在不同目录下的“别名”。|
|**软连接**|1. 快捷方式（如`/usr/bin/python`链接到`/usr/bin/python3.8`）；  <br>2. 跨文件系统访问文件；  <br>3. 动态库版本管理（如`libc.so.6`链接到`libc-2.31.so`）；  <br>4. 链接目录（如`/home/user/docs`链接到`/mnt/disk/docs`）。|

## 三、总结与高频面试问题

### 1. 核心总结

- **操作系统基础**：定义（资源管理者）、特征（并发/共享/虚拟/异步）、发展（批处理→分时→实时→现代）、体系结构（单体/分层/微内核/客户-服务器）。
    
- **软连接与硬连接**：硬连接共享inode（不能跨文件系统/链接目录），软连接是独立文件（存储路径，可跨文件系统/链接目录），目标删除后硬连接无影响、软连接失效。
    

### 2. 高频面试问题

1. **操作系统的四大特征是什么？**
    
    答：并发（宏观同时执行）、共享（资源共同使用）、虚拟（物理资源抽象为逻辑资源）、异步（执行速度不确定但结果可预测）。
    
2. **硬连接和软连接的区别？**
    
    答：硬连接共享inode（不能跨文件系统/链接目录，目标删除无影响）；软连接是独立文件（存储路径，可跨文件系统/链接目录，目标删除后失效）。
    
3. **为什么硬连接不能跨文件系统？**
    
    答：不同文件系统的inode号独立，硬连接依赖inode号，无法指向其他文件系统的inode。
    
4. **软连接的目标文件删除后，软连接会怎样？**
    
    答：软连接变为“断链”，访问时提示“文件不存在”（需重新创建或修复目标文件）。
    
5. **硬连接的链接计数是什么？删除文件时如何变化？**
    
    答：链接计数是inode中记录的文件名数量（初始为1），删除文件时计数-1，计数为0时释放数据块。
    



# 操作系统面试八股全面总结与常见面试题

## 一、操作系统核心框架

操作系统（OS）是管理硬件资源、提供用户接口的系统软件，核心围绕**进程管理、内存管理、IO管理、文件系统**四大模块展开，辅以**内核结构、系统调用、安全机制**等基础支撑。

## 二、核心模块知识点与常见面试题

### **模块1：系统结构与内核**

#### **核心知识点**

1. **内核定义**：OS核心，运行于内核态（Ring 0），管理CPU、内存、IO、进程调度，提供系统调用接口。
    
2. **内核结构类型**：
    
    - **宏内核**（Linux/Windows）：功能集中（进程、内存、驱动），性能好但稳定性差；
        
    - **微内核**（Minix/QNX）：仅保留核心功能（进程通信、调度），服务运行于用户态，稳定性好但性能低；
        
    - **混合内核**（macOS XNU）：结合两者，平衡性能与稳定性。
        
3. **用户态vs内核态**：
    
    - **用户态**（Ring 3）：应用程序运行，权限受限（不可直接访问硬件）；
        
    - **内核态**（Ring 0）：内核运行，可执行所有指令（访问硬件、修改页表）；
        
    - **切换触发**：系统调用、中断、异常；**切换开销**：保存/加载上下文、缓存失效。
        

#### **常见面试题**

1. **宏内核与微内核的区别？**
    
    答：宏内核功能集中（性能好、稳定性差），微内核仅保留核心功能（稳定性好、性能低）。
    
2. **用户态和内核态的切换流程？**
    
    答：触发（系统调用/中断）→ 保存用户上下文 → 内核处理（查系统调用表）→ 恢复用户上下文。
    
3. **为什么划分用户态和内核态？**
    
    答：安全（防用户破坏系统）、稳定（用户崩溃不影响内核）、资源管理（内核统一调度）。
    

### **模块2：内存管理**

#### **核心知识点**

1. **物理内存vs虚拟内存**：
    
    - **物理内存**：硬件DRAM，容量有限（GB级），CPU直接访问；
        
    - **虚拟内存**：逻辑扩展（GB/TB级），通过MMU映射物理内存+磁盘交换空间，实现进程隔离、按需调页。
        
2. **分段vs分页**：
    ![[Screenshot_2026-03-13-19-20-29-578_com.tencent.hunyuan.app.chat.png]]
    
3. **多级页表与TLB**：
    
    - **多级页表**（如二级页表）：解决页表空间爆炸（仅存活跃页表项），但地址转换需多次访存；
        
    - **TLB（快表）**：缓存近期页表映射（虚拟页号→物理页框号），加速地址转换（命中率依赖局部性）。
        
4. **交换空间与缺页中断**：
    
    - **交换空间**：磁盘区域，存放换出的内存页；
        
    - **缺页中断**：访问的页不在内存时触发，流程：保护现场→检查合法性→调页（读磁盘）→更新页表→恢复执行。
        
5. **页面置换算法**：
    
|算法|思想|优点|缺点|应用|
|:--|:--|:--|:--|:--|
|FIFO|换出最早进入的页|实现简单|Belady异常（页框↑缺页率↑）|几乎不用|
|OPT|换出未来最久不用的页|缺页率最低|无法预知未来|理论基准|
|LRU|换出最近最久未用的页|缺页率低|硬件实现成本高|数据库缓存|
|Clock|环形扫描+访问位（二次机会）|近似LRU，成本低|不如LRU精确|Linux（近似LRU）|
#### **常见面试题**

1. **为什么需要虚拟内存？**
    
    答：扩展物理内存、进程隔离、简化编程、支持按需调页。
    
2. **分页和分段的区别？**
    
    答：分页是物理单位（固定大小，无外部碎片），分段是逻辑单位（可变大小，无内部碎片）；分页一维地址，分段二维地址。
    
3. **TLB的作用是什么？命中率影响因素？**
    
    答：缓存页表映射加速地址转换；影响因素：TLB容量、程序局部性（局部性好则命中率高）。
    
4. **缺页中断的处理流程？**
    
    答：保护现场→检查合法性→调页（读磁盘）→更新页表→恢复执行。
    
5. **LRU算法的实现难点？Clock算法如何近似？**
    
    答：LRU需记录访问时间（硬件成本高）；Clock用环形链表+访问位，扫描时给“第二次机会”，近似LRU。
    

### **模块3：IO管理**

#### **核心知识点**

1. **IO模型**：
    
    - **阻塞IO**：IO未就绪时进程阻塞（简单但低效）；
        
    - **非阻塞IO**：IO未就绪时立即返回（需轮询，CPU利用率低）；
        
    - **IO多路复用**：单进程监控多IO描述符（select/poll/epoll），**同步IO**（数据复制需进程参与）；
        
    - **异步IO（AIO）**：内核完成数据复制后通知进程（无需参与复制）。
        
2. **IO多路复用对比**：
    
|维度|select|poll|epoll|
|:--|:--|:--|:--|
|数据结构|fd_set（位图，1024限制）|pollfd数组（无限制）|红黑树+就绪链表（动态扩容）|
|时间复杂度|O(n)（遍历所有描述符）|O(n)（遍历所有描述符）|O(1)（仅返回就绪描述符）|
|触发模式|仅LT（水平触发）|仅LT|LT/ET（边缘触发更高效）|
|适用场景|少量描述符（<1024）|中等描述符|高并发（百万级连接）|
#### **常见面试题**

1. **阻塞IO和非阻塞IO的区别？**
    
    答：阻塞IO未就绪时进程阻塞；非阻塞IO立即返回，需轮询检查。
    
2. **select/poll/epoll的区别？**
    
    答：select用固定位图（O(n)、1024限制），poll用数组（O(n)、无限制），epoll用红黑树+就绪链表（O(1)、支持ET/LT）。
    
3. **epoll的LT和ET模式区别？**
    
    答：LT模式下就绪事件反复通知（易编程）；ET模式仅通知一次（需非阻塞IO+循环读写，效率高）。
    

### **模块4：进程管理**

#### **核心知识点**

1. **并行vs并发**：
    
    - **并行**：同一时刻物理同时执行（多核）；**并发**：同一时间段交替执行（单核切换）；并行是并发的子集。
        
2. **进程状态**：五状态（创建→就绪→运行→阻塞→终止），扩展状态（挂起：就绪挂起/阻塞挂起）。
    
3. **特殊进程**：
    
    - **孤儿进程**：父进程先终止，被init（PID=1）收养；
        
    - **僵尸进程**：子进程终止后父进程未回收PCB（占PID资源），需父进程wait()或杀死父进程；
        
    - **守护进程**：后台服务（脱离终端，父进程为init，如sshd）。
        
4. **进程通信（IPC）**：
    
|方式|原理|特点|场景|
|:--|:--|:--|:--|
|共享内存|多进程映射同一物理内存|最快（无拷贝），需同步|大数据交换（数据库缓存）|
|消息队列|内核维护消息链表，send/receive|异步安全，有拷贝开销|日志收集|
|管道（匿名）|半双工，父子进程继承fd（如`ls|grep`）|字节流，容量有限（4KB）|
|信号量|计数器+P/V操作（互斥/资源计数）|低级同步（无数据传输）|生产者-消费者模型|
|套接字|TCP/UDP跨主机通信（IP+端口）|通用（本地/远程），开销大|分布式系统|

1. **进程调度算法**：
    
|算法|思想|优点|缺点|场景|
|:--|:--|:--|:--|:--|
|FCFS|先到先服务|简单公平|长作业等待久，平均等待时间长|批处理系统|
|SJF/SRTF|短作业优先（抢占/非抢占）|平均等待时间最短|需预知时间，长作业饥饿|批处理（已知时间）|
|优先级调度|高优先级优先|灵活（按需求设优先级）|低优先级饥饿（需老化）|实时系统|
|RR|时间片轮转|公平响应快|时间片敏感（开销与延迟平衡）|分时系统（Linux CFS基础）|
|MFQ（多级反馈队列）|多级队列+反馈（升级/降级）|综合最优（兼顾长短作业）|实现复杂|通用OS（Unix/Linux）|
#### **常见面试题**

1. **并行和并发的区别？**
    
    答：并行是物理同时（多核），并发是逻辑同时（单核切换）；并行是并发的子集。
    
2. **僵尸进程和孤儿进程的区别？**
    
    答：僵尸进程是子进程终止后父进程未回收（占PCB）；孤儿进程是父进程先终止（被init收养，无害）。
    
3. **进程通信中共享内存和消息队列的区别？**
    
    答：共享内存直接读写（无拷贝，需同步）；消息队列通过内核缓冲（有拷贝，异步安全）。
    
4. **为什么多级反馈队列（MFQ）是通用OS首选？**
    
    答：兼顾短作业（高优先级快速执行）、长作业（低优先级大时间片）、IO密集型（IO后升级），综合性能最优。
    

### **模块5：线程管理**

#### **核心知识点**

1. **线程vs进程**：
    
|维度|进程（资源容器）|线程（调度单位）|
|:--|:--|:--|
|资源拥有|独立地址空间、PCB、文件描述符|共享进程资源，仅私有栈、寄存器、TCB|
|切换开销|大（需切换地址空间、资源）|小（仅切换线程上下文，无地址空间切换）|
|通信方式|IPC（共享内存、消息队列等）|直接读写共享内存（需同步）|
1. **线程实现方式**：
    
    - **用户级线程（ULT）**：用户库管理（切换快，无法利用多核，阻塞影响进程）；
        
    - **内核级线程（KLT）**：内核管理（切换慢，可利用多核，阻塞不影响进程）；
        
    - **混合（M:N）**：ULT映射到KLT（如Go GMP模型，平衡速度与并行性）。
        
4. **线程同步机制**：互斥锁（临界区）、条件变量（等待条件）、信号量（计数资源）、读写锁（读多写少）、原子操作（无锁简单变量）、屏障（多线程同步点）。
    
5. **死锁**：
    
    - **四个必要条件**：互斥、占有并等待、不可剥夺、循环等待；
        
    - **处理方法**：预防（破坏条件，如按顺序申请资源）、避免（银行家算法）、检测恢复（终止线程/剥夺资源）。
        
6. **活锁vs饥饿**：
    
    - **活锁**：线程未被阻塞但不断重试无效操作（如两人互相让路）；
        
    - **饥饿**：线程长期得不到资源（如低优先级线程被抢占），解决：老化机制（优先级随时间提升）。
        

#### **常见面试题**

1. **线程和进程的区别？**
    
    答：线程共享进程资源（切换开销小，调度单位）；进程拥有独立资源（隔离性好，资源容器）。
    
2. **用户级线程和内核级线程的区别？**
    
    答：ULT用户库管理（切换快、无法利用多核）；KLT内核管理（切换慢、可利用多核）。
    
3. **死锁的四个必要条件？如何破坏？**
    
    答：互斥（资源共享）、占有并等待（预先分配资源）、不可剥夺（允许抢占）、循环等待（按顺序申请资源）。
    
4. **活锁和饥饿的区别？**
    
    答：活锁是不断重试不进展（原地踏步）；饥饿是长期得不到资源（饿死），原因是优先级或分配不均。
    

### **模块6：文件系统与链接**

#### **核心知识点**

1. **inode与目录项**：
    
    - **inode**：存储文件元数据（权限、大小、数据块指针），唯一inode号（同一文件系统内）；
        
    - **目录项**：文件名→inode号映射。
        
2. **软连接vs硬连接**：
    
|维度|硬连接（同一inode）|软连接（独立文件，存路径）|
|:--|:--|:--|
|inode|与原文件共享（号相同）|独立inode（号不同）|
|跨文件系统|不支持（inode号独立）|支持（路径可跨文件系统）|
|链接目录|不支持（防循环链接）|支持|
|目标删除影响|无（链接计数-1，计数为0才删数据）|失效（断链）|
#### **常见面试题**

1. **硬连接和软连接的区别？**
    
    答：硬连接共享inode（不能跨文件系统/链接目录，目标删除无影响）；软连接是独立文件（存储路径，可跨文件系统/链接目录，目标删除后失效）。
    
2. **为什么硬连接不能跨文件系统？**
    
    答：不同文件系统inode号独立，硬连接依赖inode号。
