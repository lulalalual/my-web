数据结构是在回答一个问题：数据应该怎么摆放，才能让后续操作更方便。数组、链表、栈、队列、树、图、哈希表都不是孤立概念，它们是针对不同操作习惯设计出来的工具。

## 数组适合按位置快速访问

数组把元素连续放在内存里，所以通过下标访问非常快。你要第 5 个元素，计算地址就能直接找到。

~~~cpp
std::vector<int> nums{10, 20, 30};
std::cout << nums[1] << std::endl;
~~~

数组的缺点是中间插入和删除麻烦。比如在开头插入一个元素，后面的元素都要往后挪。它适合读多、随机访问多、尾部追加多的场景。

## 链表适合在已知位置插入删除

链表的元素不要求连续，每个节点保存数据和下一个节点的地址。只要已经找到某个位置，插入和删除只需要改指针。

链表的代价是不能快速按下标访问。要找第 100 个节点，只能从头一步步走过去。链表也不如数组缓存友好，因为节点可能分散在内存各处。

学习链表时，重点不是背代码模板，而是画指针变化。删除节点前要保存下一个节点，反转链表时要同时维护前驱、当前和后继。

## 栈和队列是受限制的线性结构

栈的规则是后进先出，像一摞盘子，最后放上去的最先拿走。函数调用、括号匹配、单调栈都用到了这个思想。

队列的规则是先进先出，像排队买票，先来的人先处理。任务队列、广度优先搜索、消息缓冲都适合用队列。

这两种结构的价值在于限制操作方式。限制越明确，代码越不容易乱。

## 哈希表用空间换速度

哈希表通过哈希函数把 key 映射到数组位置。理想情况下，查找、插入、删除都接近 <code>O(1)</code>。

~~~cpp
std::unordered_map<std::string, int> count;
count["apple"]++;
~~~

哈希表常用于计数、去重、快速判断是否存在。它的风险是哈希冲突和扩容成本。工程里还要注意自定义 key 时，需要同时提供哈希函数和相等判断。

## 树用于表达层级和有序关系

树像公司组织架构，一个节点下面可以有多个子节点。二叉树每个节点最多两个孩子，二叉搜索树要求左边小、右边大，堆要求父节点和子节点满足大小关系。

二叉搜索树适合有序查找，但如果退化成链表，性能会变差。平衡树通过旋转保持高度，数据库索引常用的 B+ 树则是为磁盘 IO 优化过的多叉树。

学习树时，递归是最自然的工具。因为一棵树的左子树和右子树仍然是树。

~~~cpp
int countNodes(TreeNode* root) {
    if (root == nullptr) return 0;
    return 1 + countNodes(root->left) + countNodes(root->right);
}
~~~

这段代码的关键不是公式，而是把大问题拆成“当前节点 + 左子树 + 右子树”。

## 堆用来快速拿最大或最小

堆是一种特殊的完全二叉树。最大堆能快速取最大值，最小堆能快速取最小值。优先队列通常基于堆实现。

~~~cpp
std::priority_queue<int> pq;
pq.push(3);
pq.push(10);
pq.push(5);
std::cout << pq.top();
~~~

堆适合“不断加入元素，又不断取最优元素”的场景，比如任务调度、Top K、Dijkstra 算法。

## 图用来描述关系网络

图由点和边组成。社交关系、地图路线、依赖关系、网络拓扑都可以用图表示。图可以有向，也可以无向；边可以有权重，也可以没有。

图的遍历主要有 DFS 和 BFS。DFS 像沿着一条路走到底再回头，适合找连通性、拓扑结构、回溯搜索。BFS 像一层一层扩散，适合找无权图最短路径。

图题的第一步通常不是写代码，而是把关系建出来。邻接表适合稀疏图，邻接矩阵适合点少且查询边是否存在很频繁的场景。

## 并查集管理分组关系

并查集适合处理“两个元素是否属于同一组”的问题。它支持合并集合和查询代表元素。路径压缩和按秩合并可以让操作接近常数时间。

可以把并查集想成班级分组。每个人先自己一组，合并时把两组归到同一个组长下面，查询时看两个人最终组长是否相同。

## 选择数据结构的思路

不要按名字选数据结构，要按操作选。需要按位置访问，用数组。需要快速判断存在，用哈希表。需要先进先出，用队列。需要层级关系，用树。需要关系网络，用图。需要动态合并分组，用并查集。

真正掌握数据结构，是看到业务需求时能说出“数据怎么放，为什么这样放，主要操作的复杂度是多少”。
