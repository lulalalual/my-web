计算机网络可以理解成“数据包旅行”。你在浏览器输入网址后，数据要找到服务器、建立连接、发送请求、拿到响应，再一路回到你的电脑。

## 计网先说人话

网络分层不是为了背名字，而是为了分工。

~~~text
应用层：你要表达什么，比如 HTTP
传输层：怎么把数据送到进程，比如 TCP/UDP
网络层：怎么找到机器，比如 IP
链路层：怎么在相邻设备之间传，比如以太网
~~~

每层只关心自己的职责，整体通信才不会乱。

## 输入网址后发生什么

可以按这条线讲：

~~~text
输入 URL -> DNS 查 IP -> TCP 握手 -> TLS 握手 -> HTTP 请求 -> 服务器处理 -> HTTP 响应 -> 浏览器渲染
~~~

这条链路能串起大部分高频考点。

## DNS：把域名翻译成 IP

人更容易记域名，机器要靠 IP 找对方。DNS 就是把域名解析成 IP 的系统。

如果本地缓存、浏览器缓存、系统缓存里没有，就会向 DNS 服务器递归或迭代查询。

## TCP：可靠传输

TCP 的目标是可靠、有序、不丢不重地传数据。

它通过序号、确认、重传、滑动窗口、拥塞控制来做到这些。

## 三次握手

三次握手不是仪式感，而是双方确认“我能发、我能收、你也能发、你也能收”。

~~~text
客户端：我想连，你听得到吗？
服务端：我听到了，我也准备好了，你听得到我吗？
客户端：我听到了，开始通信。
~~~

如果只有两次，服务端无法确认客户端能收到自己的响应。

## 四次挥手

TCP 是全双工的，双方都可以发送数据。关闭连接时，两边的发送方向要分别关闭，所以通常是四次。

人话理解：我这边说完了，不代表你那边也说完了。

## TCP 和 UDP 区别

TCP 有连接、可靠、有序，适合文件、网页、数据库、登录等场景。

UDP 无连接、开销小、延迟低，但不保证可靠，适合语音、视频、游戏实时同步等能容忍丢包的场景。

## 流量控制和拥塞控制

流量控制是保护接收方：你别发太快，我接不过来。

拥塞控制是保护网络：网络已经堵了，大家都慢一点。

一个看接收端能力，一个看网络整体状况。

## HTTP 是什么

HTTP 是应用层协议，规定客户端和服务端怎么表达请求和响应。

~~~http
GET /posts HTTP/1.1
Host: example.com
~~~

请求里有方法、路径、头部和可能的请求体。响应里有状态码、头部和响应体。

## 常见状态码

- 200：成功。
- 301/302：重定向。
- 400：请求有问题。
- 401/403：未登录或没权限。
- 404：资源不存在。
- 500：服务器内部错误。

状态码不是装饰，它能帮前后端快速定位问题。

## HTTPS

HTTPS = HTTP + TLS。它解决三件事：加密、防篡改、身份认证。

证书用来证明服务器身份，密钥协商用来生成对称加密密钥。后续数据主要用对称加密传输，因为效率更高。

## Cookie、Session、Token

HTTP 本身是无状态的。服务端需要知道“你是谁”，就要有状态方案。

Cookie 是浏览器保存的一小段数据。Session 通常存在服务端，Cookie 里放 session id。Token 则常把身份信息签名后交给客户端保存。

## 缓存

缓存的目标是少请求、少传输、少计算。

强缓存命中时，浏览器可以不问服务器。协商缓存会问服务器资源有没有变，没变就返回 304。

## CDN

CDN 是把静态资源放到离用户更近的节点。用户访问图片、脚本、样式时，可以从附近节点获取，速度更快。

## 常见坑

- 只背 OSI 七层，不会解释真实访问链路。
- 把 TCP 可靠理解成“永远不丢”，实际上它是通过重传等机制提高可靠性。
- 不区分流量控制和拥塞控制。
- HTTPS 只说加密，不说身份认证和防篡改。
- Cookie、Session、Token 混在一起。

## 面试怎么说

如果问“输入 URL 后发生什么”，不要一口气背一大段。按 DNS、TCP、TLS、HTTP、服务器处理、浏览器渲染的顺序讲，每一步补一两句关键点。

如果问 TCP 为什么三次握手，重点说双方收发能力都要确认，避免历史连接和半连接造成问题。

## 总结

计网不是背协议名，而是理解数据怎么从一个进程到另一个进程。抓住“寻址、连接、传输、应用语义、安全和缓存”这几个关键词，大部分问题都能串起来。

### 1. 核心定义

**计算机网络体系结构**是为实现网络通信而设计的**分层模型**，将复杂的通信过程分解为若干层次，每层专注于特定功能，层间通过**接口**通信，通过**协议**（规则集合）规范交互。

**核心目标**：标准化通信流程、降低设计复杂度、便于故障定位和协议扩展。

### 1. 模型简介

> 说人话：这一节讲的是“1. 模型简介”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

**OSI（Open Systems Interconnection）七层模型**是国际标准化组织（ISO）提出的**理论参考模型**，从下到上分为7层，每层定义明确的功能和协议，强调“开放系统互联”。

### 2. 各层功能、协议与设备（从下到上）

> 说人话：这一节讲的是“2. 各层功能、协议与设备（从下到上）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

|**层级**|**核心功能**|**关键协议/技术**|**典型设备**|**数据单位**|
|:--|:--|:--|:--|:--|
|**1. 物理层**|传输**比特流**（0/1电信号/光信号），定义电气、机械、时序接口（如电压、接口形状）。|IEEE 802.3（以太网）、RS-232、USB、光纤通信|集线器（Hub）、中继器、网卡|比特（Bit）|
|**2. 数据链路层**|将比特流封装为**帧**，实现相邻节点间的可靠传输（差错控制、流量控制），通过**MAC地址**寻址。|Ethernet（以太网）、PPP（点到点协议）、HDLC、MAC地址（48位物理地址）|交换机（Switch）、网桥（Bridge）|帧（Frame）|
|**3. 网络层**|实现**跨网络的主机寻址与路由**（选择最佳路径），处理拥塞控制。|IP（IPv4/IPv6）、ICMP（ping/traceroute）、ARP（IP→MAC）、RIP/OSPF（路由协议）|路由器（Router）|分组/数据包（Packet）|
|**4. 传输层**|提供**端到端的可靠/不可靠传输**（进程间通信），实现流量控制、差错恢复。|TCP（可靠、面向连接）、UDP（不可靠、无连接）、端口号（16位，区分进程）|网关（Gateway，部分功能）|段（Segment，TCP）/数据报（Datagram，UDP）|
|**5. 会话层**|管理**通信会话**（建立、维护、终止），如断点续传、会话同步。|NetBIOS、RPC（远程过程调用）|无独立设备（软件实现）|会话数据单元（SPDU）|
|**6. 表示层**|处理**数据格式转换**（加密/解密、压缩/解压、编码转换，如ASCII→Unicode）。|JPEG（图像）、MPEG（视频）、SSL/TLS（加密）|无独立设备（软件实现）|表示数据单元（PPDU）|
|**7. 应用层**|为用户应用提供**网络服务接口**（如网页浏览、文件传输）。|HTTP（网页）、FTP（文件传输）、DNS（域名解析）、SMTP（邮件发送）、DHCP（动态IP分配）|无独立设备（应用程序）|应用数据单元（APDU）|

- **理论先行**：先定义模型，后开发协议（理想化，实际中未完全遵循）；
    
- **严格分层**：每层功能独立，层间通过接口通信，下层为上层提供服务；

### 2. 高频面试问题

> 说人话：这一节讲的是“2. 高频面试问题”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

1. **OSI七层模型每层的功能和协议？**
    
    答：物理层（比特流，以太网）、数据链路层（帧/MAC，Ethernet）、网络层（IP/路由，IP/ICMP）、传输层（端到端，TCP/UDP）、会话层（会话管理，NetBIOS）、表示层（数据格式，JPEG/SSL）、应用层（用户接口，HTTP/FTP）。

    答：网络接口层（OSI物理+数据链路）、网际层（OSI网络层）、传输层（OSI传输层）、应用层（OSI会话+表示+应用层）。
    
3. **五层结构模型每层的作用？**
    
    答：物理层（比特传输）、数据链路层（帧/MAC）、网络层（IP/路由）、传输层（TCP/UDP）、应用层（用户服务）。
    
4. **各层典型设备？**

## 一、TCP与UDP的核心区别（面试高频）

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

### 1. 基础特性对比

> 说人话：这一节讲的是“1. 基础特性对比”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

|**维度**|**TCP（Transmission Control Protocol）**|**UDP（User Datagram Protocol）**|
|:--|:--|:--|
|**连接性**|**面向连接**（三次握手建立连接，四次挥手释放连接）|**无连接**（发送前无需建立连接，直接发送数据报）|
|**可靠性**|**可靠传输**（保证数据无丢失、无重复、按序到达）|**不可靠传输**（尽最大努力交付，可能丢包、乱序、重复）|
|**传输方式**|**字节流传输**（无边界，如水管流水，需应用层分包）|**数据报传输**（有边界，每个UDP包独立，如快递包裹）|
|**首部开销**|固定20字节（无选项时），含序列号、确认号、窗口大小等|固定8字节（源端口、目的端口、长度、校验和）|
|**流量/拥塞控制**|支持（滑动窗口、慢启动、拥塞避免、快重传/快恢复）|不支持（无控制机制，可能淹没接收方）|
|**双工性**|全双工（双方可同时收发）|全双工（双方可同时收发）|
|**适用场景**|可靠性要求高（文件传输FTP、邮件SMTP、网页HTTP/HTTPS）|实时性要求高（音视频通话、直播、游戏、DNS、物联网传感器）|

### 2. 核心机制差异详解

> 说人话：这一节讲的是“2. 核心机制差异详解”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

#### （1）连接性：TCP的“三次握手”与“四次挥手”

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

- **三次握手**（建立连接）：
    
    1. 客户端→服务器：SYN=1（请求连接），seq=x（初始序列号）；
        
    2. 服务器→客户端：SYN=1，ACK=1（确认），seq=y，ack=x+1；
        
    3. 客户端→服务器：ACK=1，seq=x+1，ack=y+1（连接建立）。
        
- **四次挥手**（释放连接）：
    
    1. 主动方→被动方：FIN=1（请求关闭），seq=u；
        
    2. 被动方→主动方：ACK=1，seq=v，ack=u+1（确认关闭请求，继续发送剩余数据）；
        
    3. 被动方→主动方：FIN=1，ACK=1，seq=w，ack=u+1（数据发送完毕，请求关闭）；
        
    4. 主动方→被动方：ACK=1，seq=u+1，ack=w+1（确认关闭，等待2MSL后释放连接）。
        

**UDP无连接**：发送方直接构造UDP数据报（含源/目的端口、长度、校验和、数据），通过网络层发送，无需协商。

#### （2）可靠性：TCP的“确认+重传+排序” vs UDP的“尽力而为”

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

- **TCP可靠性机制**：
    
    - **序列号与确认号**：为每个字节编号，接收方通过ACK确认已接收的最大序列号（累计确认）；
        
    - **超时重传**：发送方未收到ACK，超时后重传数据（自适应超时时间）；
        
    - **滑动窗口**：动态调整发送速率（流量控制），避免接收方缓冲区溢出；
        
    - **拥塞控制**：通过慢启动、拥塞避免等算法感知网络拥塞，降低发送速率；
        
    - **数据校验**：首部含校验和，检测数据错误（错误则丢弃并等待重传）。
        
- **UDP不可靠性**：
    
    - 无序列号、确认号、重传机制，丢包后不通知发送方；
        
    - 无流量/拥塞控制，可能突发大量数据导致网络拥堵；
        
    - 校验和可选（IPv4中可选，IPv6中强制），错误数据可能被静默丢弃。
        

#### （3）传输方式：字节流 vs 数据报

> 说人话：这一节讲的是“（3）传输方式：字节流 vs 数据报”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **TCP字节流**：数据无边界，如发送方分3次发送“ABC”“DEF”“GHI”，接收方可能一次性接收“ABCDEFGHI”（需应用层定义边界，如HTTP的Content-Length）；
    
- **UDP数据报**：每个UDP包独立，发送方发3个包，接收方需分3次接收（包大小受MTU限制，通常≤1500字节）。
    

## 二、UDP如何保证消息不丢失？（应用层补偿机制）

### 1. 核心前提：UDP本身不保证可靠性

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

UDP设计目标是**简单、高效、低延迟**（首部仅8字节，无连接开销），其“不可靠”是刻意为之（如实时音视频允许少量丢包，优先保证流畅性）。**消息不丢失需通过应用层协议或机制实现**。

### 2. 应用层保障可靠性的常见手段

> 说人话：这一节讲的是“2. 应用层保障可靠性的常见手段”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

#### （1）确认机制（ACK）与重传机制

> 说人话：这一节讲的是“（1）确认机制（ACK）与重传机制”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **原理**：接收方收到UDP包后，向发送方返回**确认包（ACK）**；发送方未收到ACK，则**超时重传**。
    
- **实现要点**：
    
    - 为每个UDP包分配**唯一序列号**（如递增ID），接收方ACK中携带序列号；
        
    - 发送方维护“已发送未确认”队列，超时（如200ms）后重传最早未确认的包；
        
    - 接收方需去重（通过序列号），避免重复包影响。
        
- **示例**：TFTP（简单文件传输协议）基于UDP，通过“DATA包+ACK+超时重传”实现可靠传输。
    

#### （2）校验和与错误检测

> 说人话：这一节讲的是“（2）校验和与错误检测”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **UDP首部校验和**：16位校验和（覆盖首部+数据），检测传输中的比特错误（如噪声导致的数据翻转）。
    
- **应用层增强**：对关键数据（如金融交易）增加**CRC32/MD5校验**，接收方校验失败则丢弃并请求重传。
    

#### （3）序列号与乱序处理

> 说人话：这一节讲的是“（3）序列号与乱序处理”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **原理**：发送方为每个包添加**序列号**，接收方按序列号排序，缺失序列号的包请求重传。
    
- **实现要点**：
    
    - 序列号范围（如32位整数，避免溢出）；
        
    - 接收方维护“期望序列号”，缓存乱序到达的包（如视频帧），按序提交给应用层。
        

#### （4）流量控制与拥塞控制

> 说人话：这一节讲的是“（4）流量控制与拥塞控制”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **流量控制**：接收方通过ACK告知发送方可接收的最大数据量（如“窗口大小”），发送方不超过该值（避免接收方缓冲区溢出）。
    
- **拥塞控制**：发送方监测网络延迟/丢包率（如连续3个包超时），主动降低发送速率（如减半窗口），避免加剧网络拥堵。
    

#### （5）使用可靠UDP变种协议

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

为解决UDP不可靠问题，工业界设计了**可靠UDP协议（RUDP）**，在UDP基础上增加可靠性机制：

- **QUIC（Quick UDP Internet Connections）**：Google提出，集成TCP的可靠性（ACK、重传、拥塞控制）与UDP的高效（0-RTT握手），HTTP/3的底层协议；
    
- **RUDP（Reliable UDP）**：自定义可靠传输（如游戏引擎常用，保证操作指令不丢失）；
    
- **UDT（UDP-based Data Transfer Protocol）**：面向高速网络的可靠传输协议。

## 二、TCP四次挥手（释放连接）

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

TCP是**全双工通信**（双方可同时收发），需双方分别关闭连接（先关闭的一方称“主动方”，后关闭称“被动方”）。

|2|服务器→客户端|ACK=1（确认）|v（服务器最后字节序列号+1）|u+1（确认客户端FIN）|服务器：CLOSE_WAIT；客户端：FIN_WAIT_2|服务器确认收到关闭请求，继续发送剩余数据（半关闭状态）|
|3|服务器→客户端|FIN=1，ACK=1|w（服务器最后字节序列号+1）|u+1（确认客户端FIN）|服务器：LAST_ACK；客户端：FIN_WAIT_2|服务器数据发送完毕，请求关闭连接|
|4|客户端→服务器|ACK=1（确认）|u+1（客户端序列号）|w+1（确认服务器FIN）|客户端：TIME_WAIT；服务器：CLOSED|客户端确认收到服务器FIN，进入TIME_WAIT（等待2MSL后关闭）|

### 3. 关键细节

> 说人话：这一节讲的是“3. 关键细节”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **TIME_WAIT状态**：主动方发送最后一个ACK后，需等待**2MSL（最长报文段寿命，通常2分钟）**，原因：
    
    1. 确保最后一个ACK被服务器收到（若丢失，服务器会重传FIN，客户端可再次回复ACK）；
        
    2. 避免旧连接的报文段干扰新连接（旧报文段在网络中滞留，2MSL后消失）。
        
- **半关闭状态**：步骤2后，服务器仍可发送数据（CLOSE_WAIT状态），客户端仅能接收（FIN_WAIT_2状态）。
    

## 三、为什么不是两次或四次握手？

> 说人话：这一节讲的是“三、为什么不是两次或四次握手？”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

### 1. 为什么不是两次握手？

> 说人话：这一节讲的是“1. 为什么不是两次握手？”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

**问题**：无法防止**历史连接的初始化**（旧的SYN报文突然到达）。

- 例：客户端发送SYN（seq=x）后超时重传，旧SYN（x）先到达服务器，服务器回复SYN+ACK（seq=y, ack=x+1）并建立连接，但此时客户端已放弃连接，导致服务器空等，浪费资源。
    
- **三次握手解决**：服务器需收到客户端的第三次ACK（确认服务器的SYN），才认为连接有效，避免历史连接干扰。

> 说人话：这一节讲的是“2. 为什么不是四次握手？”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

**问题**：三次握手已足够确认双方收发能力，四次握手会冗余。

- 步骤2中，服务器的SYN和ACK可合并为一个报文（同时发送），无需分两次（四次握手中的步骤2和3可合并，实际就是三次）。

## 七、TCP重传机制

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

### 1. 超时重传（Timeout Retransmission）

> 说人话：这一节讲的是“1. 超时重传（Timeout Retransmission）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **原理**：发送方为每个报文段设置**超时定时器**（基于RTT动态计算），若超时未收到ACK，则重传该报文段；
    
- **RTT计算**：通过采样往返时间（如首次RTT=200ms，后续加权平均），超时时间=RTT×2（保守估计）；
    
- **缺点**：超时周期长（尤其RTT大时），效率低。
    

### 2. 快重传（Fast Retransmission）

> 说人话：这一节讲的是“2. 快重传（Fast Retransmission）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **原理**：接收方收到**失序报文段**时，立即发送**重复ACK**（如期望seq=100，收到seq=102，发送ACK=100）；
    
- **触发条件**：发送方收到**3个重复ACK**（如ACK=100连续3次），判定seq=100的报文段丢失，立即重传（无需等待超时）；
    
- **优点**：比超时重传更快恢复丢包，减少延迟。

    答：SYN→SYN+ACK→ACK；三次是为了确认双方收发能力，防止历史连接干扰（两次无法区分新旧SYN）。
    
2. **TCP四次挥手的过程？TIME_WAIT状态的作用？**
    
    答：FIN→ACK→FIN→ACK；TIME_WAIT等待2MSL，确保最后一个ACK被服务器收到，避免旧报文干扰新连接。

    答：流量控制是“端到端”（接收方rwnd限制发送方），拥塞控制是“网络到端”（cwnd感知网络拥塞）。

## **四、阶段3：TCP连接建立（三次握手）**

> 说人话：网络知识先顺链路：谁发、发到哪、协议怎么保证、失败怎么处理。不要只背名词，要能放进一次真实请求里理解。

获取IP地址后，浏览器通过**TCP协议**（传输控制协议）与服务器建立**可靠的端到端连接**，确保数据不丢失、不重复。三次握手流程如下：

|**步骤**|**方向**|**标志位**|**序列号（seq）**|**确认号（ack）**|**状态变化**|**核心作用**|
|:--|:--|:--|:--|:--|:--|:--|
|1|客户端→服务器|SYN=1（同步请求）|x（客户端ISN）|0（无效）|客户端：`SYN_SENT`|客户端请求建立连接，告知服务器初始序列号x|
|2|服务器→客户端|SYN=1，ACK=1（确认）|y（服务器ISN）|x+1（确认客户端SYN）|服务器：`SYN_RCVD`；客户端：`SYN_SENT`|服务器同意连接，告知客户端初始序列号y，确认收到客户端SYN|
|3|客户端→服务器|ACK=1（确认）|x+1（客户端序列号+1）|y+1（确认服务器SYN）|客户端：`ESTABLISHED`；服务器：`ESTABLISHED`|客户端确认收到服务器SYN，连接正式建立|

**关键说明**：

- **ISN（初始序列号）**：随机生成（非固定1），防止攻击者猜测序列号伪造数据；
    
- **状态变化**：`SYN_SENT`（客户端等待服务器响应）→`SYN_RCVD`（服务器等待客户端确认）→`ESTABLISHED`（连接建立）。

## 一、IP协议（Internet Protocol）

> 说人话：这一节讲的是“一、IP协议（Internet Protocol）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

### 1. 核心定义与作用

> 说人话：这一节讲的是“1. 核心定义与作用”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

**IP协议**是**网络层**的核心协议（TCP/IP模型的网际层，OSI模型的网络层），负责**跨网络的主机寻址、路由选择与数据报传输**，是互联网“互联互通”的基石。

#### 核心作用：

> 说人话：这一节讲的是“核心作用”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **寻址**：通过**IP地址**唯一标识网络中的主机（如`192.168.1.1`）；
    
- **路由**：通过路由协议（如RIP/OSPF）选择数据报从源到目的的**最佳路径**；
    
- **分段与重组**：当数据报超过链路层MTU（最大传输单元，如以太网MTU=1500字节）时，IP层将其**分段**，接收方重组；
    
- **数据报传输**：以“尽最大努力交付”为原则，将数据报从源主机发送到目的主机（不保证可靠、有序、无重复）。
    

### 2. IP协议的特点

> 说人话：这一节讲的是“2. IP协议的特点”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

|**特点**|**说明**|
|:--|:--|
|**无连接**|发送数据前无需建立连接（如寄信无需提前通知收件人），每个数据报独立路由；|
|**不可靠**|不保证数据报一定到达、不保证按序到达、不保证无重复（可靠性由TCP实现）；|
|**尽最大努力交付**|仅承诺“尝试发送”，丢包、错误由上层协议（如TCP）处理；|
|**面向数据报**|数据以“IP数据报”为单位传输（类似快递包裹，每个包裹独立处理）。|

### 3. IPv4与IPv6（版本差异）

> 说人话：这一节讲的是“3. IPv4与IPv6（版本差异）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

|**维度**|**IPv4**|**IPv6**|
|:--|:--|:--|
|**地址长度**|32位（4字节），如`192.168.1.1`|128位（16字节），如`2001:0db8:85a3::8a2e:0370:7334`|
|**地址表示**|点分十进制（4组0-255）|冒分十六进制（8组4位十六进制，缩写连续0）|
|**地址数量**|约43亿（2³²），已耗尽|约3.4×10³⁸（2¹²⁸），几乎无限|

### 4. IP数据报格式（IPv4，重点字段）

> 说人话：这一节讲的是“4. IP数据报格式（IPv4，重点字段）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

IP数据报由**首部**和**数据部分**组成，首部含20字节固定部分+可选字段（最多40字节），关键字段如下：

|**字段**|**长度**|**作用**|
|:--|:--|:--|
|**版本（Version）**|4位|标识IP版本（IPv4=0100，IPv6=0110）|
|**首部长度（IHL）**|4位|首部总长度（单位4字节，最小5→20字节，最大15→60字节）|
|**服务类型（TOS）**|8位|标识服务质量（如优先级、延迟/吞吐量偏好，IPv6中改为“流量类别”）|
|**总长度（Total Length）**|16位|数据报总长度（首部+数据，最大65535字节，超过MTU需分段）|
|**标识（Identification）**|16位|分段数据的唯一标识（同一数据报的分段标识相同，用于接收方重组）|
|**标志（Flags）**|3位|分段控制（DF=不分片，MF=还有后续分段）|
|**片偏移（Fragment Offset）**|13位|分段数据在原数据报中的位置（单位8字节，如偏移100→第800字节开始）|
|**生存时间（TTL）**|8位|数据报在网络中的最大跳数（每经路由器减1，TTL=0时丢弃，防环路）|
|**协议（Protocol）**|8位|指示上层协议（如TCP=6，UDP=17，ICMP=1）|
|**首部校验和（Header Checksum）**|16位|校验首部错误（仅校验首部，数据部分由上层协议校验）|
|**源IP地址（Source IP）**|32位|发送方IP地址|
|**目的IP地址（Destination IP）**|32位|接收方IP地址|

> 说人话：这一节讲的是“（2）分段与重组”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- 数据报每经过一个路由器，TTL减1；当TTL=0时，路由器丢弃数据报并发送**ICMP超时消息**（避免环路导致网络拥塞）。
    

## 二、ARP协议（Address Resolution Protocol）

> 说人话：这一节讲的是“二、ARP协议（Address Resolution Protocol）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

### 1. 核心定义与作用

> 说人话：这一节讲的是“1. 核心定义与作用”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

**ARP协议**（地址解析协议）是**数据链路层与网络层的桥梁**，负责将**IP地址映射为MAC地址**（物理地址），实现同一局域网内主机的直接通信。

#### 核心问题：

> 说人话：这一节讲的是“核心问题”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

网络层用IP地址寻址（逻辑地址），数据链路层用MAC地址寻址（物理地址，如以太网的48位MAC）。当主机A要向主机B发送数据时，已知B的IP地址，但需知道B的MAC地址才能在链路层封装帧——ARP协议解决此问题。

### 2. ARP协议工作过程（同一局域网内）

> 说人话：这一节讲的是“2. ARP协议工作过程（同一局域网内）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

以“主机A（IP_A=192.168.1.2，MAC_A=AA-AA-AA-AA-AA-AA）向主机B（IP_B=192.168.1.3，未知MAC_B）发送数据”为例：

#### 步骤1：检查ARP缓存

> 说人话：这一节讲的是“步骤1：检查ARP缓存”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

主机A先查询本地**ARP缓存表**（存储近期IP-MAC映射，如`arp -a`命令查看），若找到IP_B对应的MAC_B，直接使用；若未找到，进入下一步。

#### 步骤2：广播ARP请求报文

> 说人话：这一节讲的是“步骤2：广播ARP请求报文”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

主机A构造**ARP请求报文**（目标MAC为广播地址`FF-FF-FF-FF-FF-FF`），通过数据链路层广播到局域网：

- **源IP**：IP_A（192.168.1.2），**源MAC**：MAC_A（AA-AA-AA-AA-AA-AA）；
    
- **目标IP**：IP_B（192.168.1.3），**目标MAC**：00-00-00-00-00-00（未知，填0）；
    
- **操作类型**：1（请求）。
    

#### 步骤3：主机B单播回复ARP响应

> 说人话：这一节讲的是“步骤3：主机B单播回复ARP响应”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

局域网内所有主机收到广播请求后，仅主机B（IP_B=192.168.1.3）识别目标IP为自己，构造**ARP响应报文**（单播发送给主机A）：

- **源IP**：IP_B（192.168.1.3），**源MAC**：MAC_B（BB-BB-BB-BB-BB-BB）；
    
- **目标IP**：IP_A（192.168.1.2），**目标MAC**：MAC_A（AA-AA-AA-AA-AA-AA）；
    
- **操作类型**：2（响应）。
    

#### 步骤4：主机A更新ARP缓存

> 说人话：这一节讲的是“步骤4：主机A更新ARP缓存”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

主机A收到响应后，将“IP_B→MAC_B”映射存入ARP缓存表（默认有效期2-10分钟），后续通信直接使用该MAC地址。

### 3. 跨网段通信时的ARP（通过网关）

> 说人话：这一节讲的是“3. 跨网段通信时的ARP（通过网关）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

若主机A（192.168.1.2）与主机B（10.0.0.2）不在同一局域网（子网掩码255.255.255.0），A需通过**网关**（如192.168.1.1）转发数据：

1. 主机A判断B的IP与自己不在同一网段（192.168.1.2 & 255.255.255.0 = 192.168.1.0，10.0.0.2 & 255.255.255.0 = 10.0.0.0）；
    
2. 主机A通过ARP获取**网关的MAC地址**（广播请求网关IP对应的MAC）；
    
3. 主机A将数据报封装为帧（目标MAC=网关MAC），发送给网关；
    
4. 网关根据路由表转发数据报到目标网段，最终通过ARP获取B的MAC地址并送达。

### 5. ARP的特殊类型

> 说人话：对象相关内容重点看生命周期。对象什么时候出生、什么时候复制或移动、什么时候销毁，决定了代码是否安全。

- **免费ARP（Gratuitous ARP）**：主机主动广播自己的IP-MAC映射（目标IP=源IP），用于**检测IP冲突**（若其他主机回应，说明IP已被占用）或**更新缓存**（如更换网卡后通知其他主机）；
    
- **代理ARP（Proxy ARP）**：路由器代替其他网段主机响应ARP请求（如主机A问网关IP的MAC，网关用自己的MAC响应，替A转发数据）；
    
- **反向ARP（RARP）**：已知MAC地址求IP地址（已被DHCP取代）。
    

## 三、IP与ARP的协同工作流程（示例）

> 说人话：这一节讲的是“三、IP与ARP的协同工作流程（示例）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

以“浏览器访问`www.example.com`（IP=93.184.216.34）”为例，IP与ARP的协同：

1. 浏览器通过DNS解析获取`www.example.com`的IP=93.184.216.34；
    
2. 主机判断该IP与自己在**同一网段**（如本机IP=192.168.1.2，子网掩码255.255.255.0，目标网段=93.184.216.0≠192.168.1.0）；
    
3. 主机通过ARP获取**网关IP（192.168.1.1）对应的MAC地址**（广播请求→网关单播响应）；
    
4. 主机将数据报封装为帧（目标MAC=网关MAC），通过IP协议发送到网关；
    
5. 网关根据路由表转发数据报到互联网，途中通过ARP获取下一跳路由器的MAC地址，最终送达目标服务器。

> 说人话：这一节讲的是“四、总结与高频面试问题”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

### 1. 核心总结

> 说人话：这一节讲的是“1. 核心总结”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **IP协议**：网络层核心，无连接、不可靠，负责寻址、路由、分段重组，IPv4（32位）向IPv6（128位）过渡；
    
- **ARP协议**：数据链路层与网络层桥梁，通过“广播请求-单播响应”将IP映射为MAC，同一局域网直接通信，跨网段通过网关；
    
- **协同工作**：IP确定目标主机，ARP获取目标MAC（或网关MAC），共同完成数据报传输。
    

### 2. 高频面试问题

> 说人话：这一节讲的是“2. 高频面试问题”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

1. **IP协议的作用和特点？**
    
    答：作用是寻址、路由、分段重组、数据报传输；特点是无连接、不可靠、尽最大努力交付、面向数据报。
    
2. **ARP协议的工作过程？**
    
    答：同一局域网内，主机A广播ARP请求（目标IP=B的IP，目标MAC=广播），B单播回复ARP响应（含B的MAC），A缓存IP-MAC映射。
    
3. **IP数据报首部的关键字段？**
    
    答：版本（IPv4/IPv6）、TTL（防环路）、协议（上层协议类型）、源/目的IP、总长度、标识/片偏移（分段重组）。
    
4. **TTL的作用？**
    
    答：数据报每经路由器TTL减1，TTL=0时丢弃，防止环路导致网络拥塞。
    
5. **跨网段通信时ARP如何工作？**
    
    答：主机通过ARP获取网关的MAC地址，将数据报发给网关，由网关转发到目标网段。

|3|客户端→服务器|验证服务器证书（CA签名合法性），生成**预主密钥**`pre_master_secret`，用服务器公钥加密后发送|加密的`pre_master_secret`|
|4|双方计算会话密钥|客户端/服务器用`client_random + server_random + pre_master_secret`生成**对称密钥**（如AES密钥）|会话密钥（仅双方知晓）|
|5|客户端→服务器|发送`Finished`消息（用会话密钥加密，验证握手成功）|加密的握手摘要|
|6|服务器→客户端|发送`Finished`消息（同上）|加密的握手摘要|

**TLS 1.3简化**：合并部分步骤，减少握手延迟（如1-RTT握手）。

#### 阶段3：加密数据传输

> 说人话：这一节讲的是“阶段3：加密数据传输”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

TLS握手成功后，双方用**对称密钥**加密应用数据（HTTP报文），通过TCP连接传输，确保机密性和完整性（用MAC校验）。

### 3. 关键技术

> 说人话：这一节讲的是“3. 关键技术”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **非对称加密**（RSA/ECC）：用于TLS握手阶段交换预主密钥（公钥加密，私钥解密）；
    
- **对称加密**（AES/ChaCha20）：用于数据传输阶段（加密效率高）；
    
- **数字证书**：由CA机构颁发，证明服务器身份（含服务器公钥、域名、有效期等）；
    
- **哈希算法**（SHA-256）：生成消息摘要，校验数据完整性。
    

## 四、Session和Cookie（会话管理）

> 说人话：这一节讲的是“四、Session和Cookie（会话管理）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

### 1. 核心定义

> 说人话：这一节讲的是“1. 核心定义”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **Cookie**：**客户端存储**的小型文本数据（4KB以内），由服务器通过`Set-Cookie`响应头设置，后续请求自动携带（通过`Cookie`请求头）。
    
- **Session**：**服务器端存储**的用户会话信息（如用户ID、权限），通过**Session ID**关联客户端（Session ID通常存储在Cookie中）。
    

### 2. 区别与关联

> 说人话：这一节讲的是“2. 区别与关联”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

|**维度**|**Cookie**|**Session**|
|:--|:--|:--|
|**存储位置**|客户端（浏览器/硬盘）|服务器端（内存/数据库/Redis）|
|**安全性**|较低（易被窃取/篡改，需`HttpOnly`/`Secure`防护）|较高（数据在服务器，仅暴露Session ID）|
|**生命周期**|可设置过期时间（`Expires`/`Max-Age`），持久化存储|随会话结束（浏览器关闭）或超时（如30分钟无操作）失效|
|**存储容量**|单个Cookie≤4KB，每个域名≤20个Cookie|无限制（取决于服务器资源）|
|**依赖关系**|Session ID通常通过Cookie传递（也可URL重写）|依赖Cookie（或URL重写）传递Session ID|

### 3. Cookie核心属性

> 说人话：这一节讲的是“3. Cookie核心属性”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- `Domain`：指定Cookie生效的域名（如`.example.com`对子域名生效）；
    
- `Path`：指定Cookie生效的路径（如`/user`仅对`/user`路径生效）；
    
- `Expires/Max-Age`：过期时间（`Max-Age=0`表示删除Cookie）；
    
- `HttpOnly`：禁止JavaScript访问（防XSS攻击）；
    
- `Secure`：仅通过HTTPS传输；
    
- `SameSite`：限制跨域请求携带Cookie（`Strict`/`Lax`/`None`）。
    

### 4. 工作流程（以登录为例）

> 说人话：这一节讲的是“4. 工作流程（以登录为例）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

1. 用户提交登录表单（POST请求，含用户名密码）；
    
2. 服务器验证通过后，生成**Session ID**（如UUID），在服务器存储Session数据（如`{sessionId: "xxx", userId: 1}`）；
    
3. 服务器通过`Set-Cookie: sessionId=xxx; HttpOnly; Secure`响应头，将Session ID写入客户端Cookie；
    
4. 后续请求（如访问`/user`）自动携带Cookie（`Cookie: sessionId=xxx`）；
    
5. 服务器通过Session ID查询Session数据，验证用户身份，返回个性化内容。
    

## 五、状态码（Status Code）

> 说人话：这一节讲的是“五、状态码（Status Code）”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

### 1. 分类与核心含义

> 说人话：对象相关内容重点看生命周期。对象什么时候出生、什么时候复制或移动、什么时候销毁，决定了代码是否安全。

HTTP状态码由3位数字组成，首位定义类别，共5类：

|**类别**|**首位**|**含义**|**核心场景**|
|:--|:--|:--|:--|
|1xx|1|信息性（临时响应）|请求已接收，继续处理（如`100 Continue`）|
|2xx|2|成功（请求正常处理）|请求成功（如`200 OK`、`201 Created`）|
|3xx|3|重定向（需进一步操作）|资源位置变更（如`301 Moved Permanently`）|
|4xx|4|客户端错误（请求有误）|请求语法错误/资源不存在（如`404 Not Found`）|
|5xx|5|服务器错误（服务器故障）|服务器处理失败（如`500 Internal Server Error`）|

|**状态码**|**短语**|**含义**|**示例场景**|

|**201**|Created|资源创建成功（如POST提交订单）|注册新用户成功|

## 六、总结与高频面试问题

> 说人话：这一节讲的是“六、总结与高频面试问题”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

### 1. 核心总结

> 说人话：这一节讲的是“1. 核心总结”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

- **GET vs POST**：GET查数据（URL参数、幂等、缓存），POST改数据（请求体、非幂等、不缓存）；
    
- **HTTP报文**：请求报文（请求行+头+空行+体），响应报文（状态行+头+空行+体）；
    
- **HTTPS**：TCP握手→TLS握手（证书验证、密钥协商）→对称加密传输（AES/RSA混合加密）；
    
- **Session/Cookie**：Cookie客户端存ID，Session服务器存数据，Session ID通过Cookie传递；
    
- **状态码**：2xx成功、3xx重定向、4xx客户端错、5xx服务器错，记住常见码（200/301/302/304/404/500）。
    

### 2. 高频面试问题

> 说人话：这一节讲的是“2. 高频面试问题”。先别急着背定义，先抓住它解决的问题、代码里怎么写、边界在哪里。

1. **GET和POST的区别？**
    
    答：语义（查vs改）、参数位置（URL vs 请求体）、安全性（暴露vs隐藏）、幂等性（幂等vs非幂等）、缓存（可缓存vs默认不缓存）、长度限制（有vs无）。
    
2. **HTTPS为什么安全？**
    
    答：通过TLS加密（非对称加密交换密钥+对称加密传输数据）、CA证书验证服务器身份、MAC校验数据完整性。
    
3. **Session和Cookie的区别？**
    
    答：存储位置（服务器vs客户端）、安全性（高vs低）、生命周期（会话级vs可持久化）、依赖关系（Session ID通过Cookie传递）。
    
4. **常见HTTP状态码及含义？**
    
    答：200（成功）、301（永久重定向）、302（临时重定向）、304（缓存有效）、404（资源不存在）、500（服务器错误）。
    
5. **HTTP请求报文包含哪些部分？**
    
    答：请求行（方法+URL+协议）、请求头（Host/User-Agent/Cookie等）、空行、请求体（POST数据）。
