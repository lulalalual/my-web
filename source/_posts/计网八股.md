计算机网络不是一堆协议名，而是在解决一个朴素问题：一台机器上的程序，怎样把数据可靠或快速地交给另一台机器上的程序。所有分层、协议、握手、加密，都是围绕这个问题展开的。

## 分层是为了把复杂问题拆开

网络分层像快递系统。应用层关心寄什么内容，传输层关心怎么从一个程序送到另一个程序，网络层关心怎么跨网络找到机器，链路层关心同一段链路里怎么传输。

HTTP 属于应用层，TCP 和 UDP 属于传输层，IP 属于网络层，以太网和 Wi-Fi 属于链路层。分层的好处是每一层只关注自己的责任，下面一层变了，上面不一定要改。

## TCP 为什么需要三次握手

TCP 建连接前，双方要确认彼此都能发送和接收。三次握手可以这样理解：客户端说“我能发，你能收吗”；服务端回答“我能收，也能发”；客户端最后确认“我也能收”。

如果只有两次握手，服务端无法确认客户端是否真的收到了自己的确认，容易建立错误连接。三次握手不是形式主义，而是在不可靠网络上建立可靠通信的最低确认过程。

## 四次挥手为什么比握手多一次

关闭连接时，两个方向的数据流要分别关闭。客户端说“我不发了”，服务端可能还有数据没发完，所以先确认；等服务端也发完了，再说“我也不发了”，客户端再确认。

这就是四次挥手看起来比三次握手多的原因。连接关闭不是一刀切，而是两个方向各自收尾。

## TCP 如何保证可靠

TCP 的可靠性来自多种机制配合。序号保证数据顺序，确认应答告诉发送方哪些数据到了，超时重传处理丢包，滑动窗口控制发送速度，拥塞控制避免把网络打爆。

可以把 TCP 想成一个谨慎的快递员：每个包都有编号，对方收到了会回执，没收到就重发，同时还会根据路况调整投递速度。

## UDP 不可靠但不等于没用

UDP 不建立连接，不保证顺序，也不保证送达。但它开销小、延迟低，适合更看重实时性的场景。比如游戏同步、直播、语音通话，可以接受少量丢包，但不能接受长时间等待。

很多基于 UDP 的系统会在业务层自己补可靠性，比如给关键消息加确认、给状态包加时间戳、允许旧包丢弃。

## HTTP 是应用层的请求响应协议

HTTP 的基本模型很简单：客户端发请求，服务端回响应。请求里有方法、路径、头和正文；响应里有状态码、头和正文。

~~~text
GET /posts HTTP/1.1
Host: example.com
~~~

GET 通常用于读取，POST 用于提交，PUT/PATCH 用于更新，DELETE 用于删除。状态码也不要死背，先按类别理解：2xx 成功，3xx 重定向，4xx 客户端问题，5xx 服务端问题。

## HTTPS 在 HTTP 外面加了安全层

HTTPS 不是新协议，而是在 HTTP 和 TCP 之间加了 TLS。它解决三个问题：加密内容，防止中间人偷看；校验证书，确认你连的是正确服务器；校验数据完整性，防止传输中被篡改。

浏览器访问 HTTPS 网站时，会先进行 TLS 握手，协商加密参数，验证证书，然后再传 HTTP 数据。

## DNS 把域名翻译成 IP

用户记域名，机器找 IP。DNS 就负责把域名解析成 IP 地址。解析可能经过浏览器缓存、操作系统缓存、本地 DNS、根域名服务器、顶级域名服务器和权威 DNS。

如果网站访问慢，DNS 也是一个可能瓶颈。工程里会用缓存、智能 DNS、CDN 等方式缩短解析和访问路径。

## CDN 把内容放近用户

CDN 的核心是把静态资源缓存到离用户更近的节点。用户访问图片、脚本、样式时，不必都回源站，直接从附近节点拿。

这也是为什么国内部署要考虑云厂商和 CDN。Vercel 在海外访问体验好，但国内访问可能受网络线路影响；如果面向国内用户，域名备案、国内 CDN 和国内对象存储会更稳定。

## 网络排查的基本顺序

遇到网络问题时，先不要乱猜。可以按顺序检查：域名能不能解析，IP 能不能连通，端口是否开放，TLS 证书是否正常，HTTP 状态码是什么，服务端日志有没有报错。

网络知识最有价值的地方，不是背出协议全称，而是能把一次请求从浏览器出发到服务器返回的全过程讲清楚。
