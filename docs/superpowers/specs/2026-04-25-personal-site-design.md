# 个人网站设计规格

## 1. 项目目标

从 0 开始搭建一个可部署的个人网站，用于：

- 展示个人项目与成果
- 长期沉淀和发布个人笔记
- 通过强视觉、强记忆点的首页建立个人品牌

该网站不是普通作品集，而是一个以 `iPhone` 为视觉母体、带 `3D` 与 `液态玻璃` 风格的互动式个人站。

## 2. 核心产品方向

网站整体采用 `Apple 精致玩具感` 的视觉方向，而不是像素游戏风。

首页打开后直接进入一个包裹在 `iPhone` 机身中的 3D 微型横版展示舞台：

- 一个玩具感角色会自动前进
- 角色会自动跨越障碍、经过场景节点、抵达项目展台
- 每个项目展台代表一个项目成就
- 展台采用 Apple 风格的卡片化 3D 物件，而不是传统像素关卡牌
- 首页默认自动播放
- 用户可以手动打断、暂停、重播或拖动进度

网站除了首页展示层，还必须包含一个支持在线编辑的笔记系统。

## 3. 技术方案

采用以下固定技术栈：

- 前端框架：`Next.js`（App Router）
- 动画与页面过渡：`Framer Motion`
- 3D 场景：`React Three Fiber`
- 数据与鉴权：`Supabase`
- 登录方式：`GitHub OAuth`
- 部署平台：`Vercel`

不采用 Phaser 作为主渲染方案。该项目定位是“3D 互动作品集网站”，不是纯游戏引擎驱动的网页游戏。

## 4. 权限与内容管理

站点分为公开展示区与登录后创作区。

### 4.1 公开访问

所有访客都可以：

- 浏览首页 3D 项目展示
- 查看项目详情页
- 浏览已发布笔记
- 查看关于信息与外部链接

### 4.2 登录后权限

只允许站点所有者通过 `GitHub 登录` 后进入后台。

权限规则固定为：

- 使用 `Supabase Auth` 接入 GitHub OAuth
- 登录成功后检查 GitHub 账号白名单
- 只有指定 GitHub 账号拥有写权限
- 非白名单账号即使登录成功，也不能进入后台编辑区

当前唯一允许编辑后台的 GitHub 账号固定为：

- `lulalalual`
- GitHub 地址：`https://github.com/lulalalual`

### 4.3 后台内容管理

后台区用于管理：

- 笔记的创建、编辑、发布、下线、删除
- 项目的标题、简介、技术栈、亮点、封面、外链
- 首页 Hero 文案、个人资料、社交链接、项目排序

## 5. 页面信息架构

### 5.1 公开页面

- `/`：首页，iPhone 风格 3D 自动通关项目展示
- `/projects`：项目总览页
- `/projects/interview-master`：计算机面试大师详情页
- `/projects/tower-defense-duo`：塔防双人详情页
- `/notes`：笔记列表页
- `/notes/[slug]`：笔记详情页

### 5.2 后台页面

- `/studio`：后台总控台
- `/studio/notes`：笔记管理页
- `/studio/notes/new`：新建笔记
- `/studio/notes/[id]`：编辑笔记
- `/studio/projects`：项目管理页
- `/studio/settings`：站点设置页

## 6. 首页视觉与交互设计

### 6.1 整体形态

首页外层是高度拟真的 `iPhone` 设备外壳，包含：

- 圆角机身
- 顶部灵动岛/听筒区域
- 底部 Home Indicator
- iOS 风格分层导航

内层是一个嵌入手机屏幕中的 2.5D / 轻 3D 舞台。

### 6.2 场景表现

首页场景必须体现以下视觉特征：

- 精致玩具感角色与障碍
- 柔和景深与空间层次
- Apple 风格的高亮、磨砂、反射与通透感
- 液态玻璃式浮层、按钮与卡片
- 不走像素风，不走重工业游戏 UI

### 6.3 动画策略

动画由两层构成：

- `React Three Fiber` 负责场景、镜头、角色路径和 3D 物件演出
- `Framer Motion` 负责 UI、卡片弹层、页面过渡和液态玻璃动效

### 6.4 首页交互

首页默认行为：

- 自动播放角色通关路径
- 自动展示项目节点
- 自动聚焦项目卡片

用户可执行的交互：

- 暂停/重播首页演出
- 手动拖动进度或切换项目节点
- 点击项目卡片进入详情页
- 切换到底部 Tab 或入口进入 Notes 模块

## 7. 项目展示内容

首页首批项目固定为以下两个：

### 7.1 计算机面试大师

展示重点：

- AI 模拟面试
- 流式交互体验
- Monaco 编辑器
- 系统设计白板
- 能力雷达与复盘能力
- React / TypeScript / Vite / Tailwind / Zustand / Express 等技术特征

页面调性应更接近高级产品介绍页，强调专业度、完整度与产品感。

### 7.2 塔防双人

展示重点：

- 双人塔防玩法
- 敌人波次与塔类型
- 放置与升级机制
- C++ / SDL2 技术实现
- 游戏资源与系统模块完整性

页面调性应更接近创意游戏展示页，但仍保持 Apple 风格统一设计系统。

## 8. 笔记系统设计

笔记系统采用 `Markdown-first` 方案。

### 8.1 编辑方式

后台编辑器以 Markdown 为主，保存两类信息：

- 结构化元数据：标题、摘要、标签、封面、状态、发布时间、slug
- 正文内容：Markdown 源文

### 8.2 前台能力

前台必须支持：

- 笔记列表浏览
- 按发布时间排序
- 标签筛选
- 搜索
- Markdown 渲染
- 代码块高亮
- 标题层级与目录展示

### 8.3 后台能力

后台必须支持：

- 新建笔记
- 编辑笔记
- 草稿保存
- 发布/取消发布
- 删除笔记
- 维护标签与摘要

## 9. 数据模型

Supabase 初始数据结构收敛为以下核心实体：

### 9.1 `profiles`

用于保存站点所有者的公开资料与 GitHub 身份映射。

核心字段建议包括：

- `id`
- `github_id`
- `github_username`
- `display_name`
- `avatar_url`
- `bio`

### 9.2 `projects`

用于保存项目内容。

核心字段建议包括：

- `id`
- `slug`
- `title`
- `summary`
- `description`
- `tech_stack`
- `highlights`
- `cover_image`
- `repo_url`
- `demo_url`
- `order_index`
- `is_published`

### 9.3 `notes`

用于保存笔记内容。

核心字段建议包括：

- `id`
- `slug`
- `title`
- `excerpt`
- `content_markdown`
- `cover_image`
- `status`
- `is_pinned`
- `published_at`
- `updated_at`

### 9.4 `note_tags`

用于保存笔记标签关系。

### 9.5 `site_settings`

用于保存首页与全站配置。

核心内容包括：

- 首页文案
- 社交链接
- 个人介绍
- 首页项目排序
- 可配置外链

## 10. 设计系统原则

全站视觉必须保持以下原则：

- 明显参考 iPhone / iOS 语言
- 用液态玻璃、磨砂、通透、景深构建层次
- 用精致玩具感和轻 3D 强化记忆点
- 避免像素游戏感、赛博朋克感和厚重游戏 HUD
- 页面仍需保证可读性、可点击性与移动端适配

## 11. 部署要求

部署方案固定为：

- 前端与 SSR/路由能力部署到 `Vercel`
- 数据库、Auth、存储部署到 `Supabase`
- 通过环境变量连接 Supabase 与 GitHub OAuth

系统上线后至少满足：

- 公开访问稳定
- GitHub 登录可用
- 白名单权限控制生效
- 后台笔记在线编辑可用
- 首页 3D 演示在桌面与移动端都可正常运行

## 12. 当前结论

本项目最终被定义为：

一个基于 `Next.js + Supabase + Vercel` 的个人网站，首页以 `iPhone` 风格的 3D 自动通关舞台展示项目，辅以 `Markdown` 驱动的在线笔记系统，并通过 `GitHub OAuth` 实现仅站点所有者可编辑的后台能力。
